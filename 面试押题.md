## 项目强相关题

- `项目定位、算法亮点和推荐闭环三部分`：产品目标是电影推荐，希望不用服务器在本地运行，没有上在线服务。算法从基础协同过滤和内容推荐出发，升级到 `Word2Vec/GloVe` 语义召回，再用本地 `PPO` 做重排；工程侧做了经验池、批量训练、模型版本管理和后台实验页，总体上是一个能迭代的推荐系统。

- `链路概括`：数据采集 -> 多路召回 -> 语义混合候选 -> PPO 重排 -> 离线评估与版本管理 -> 在线反馈再训练。

- `召回和重排链路`：召回层先做基础内容推荐、基础协同过滤，再做 `word2vec_content`、`glove_content`、`word2vec_cf`、`glove_cf` 这几路语义召回，组合成 `semantic_hybrid`。这一步主要解决候选集质量问题。重排层不是重新从全集选片，而是对候选集做 `PPO` 重排，并引入 `ε-greedy`、覆盖率奖励和多样性奖励。训练数据不会每来一条就更新，而是先进经验池，积累到一定量后批量训练。

- `为什么 PPO 放在重排层`：RL 更适合在小候选集上做组合优化，直接做全量召回训练不稳定。

- `为什么不用 Redis、Kafka、Spark、向量库`：`Flask` 足够承载页面、接口和后台管理；`SQLite` 足够承载电影数据、行为数据、经验池和模型元数据。这样开发快、部署简单；缺点是并发能力和事务吞吐有限。

- `如果迁移到工业级怎么拆`：当前是单机实验闭环；若迁移到工业级，可以把存储、特征服务和在线推断拆分出去。

- `为什么采用多路召回 + 重排`：召回层负责从大量电影里先筛出几百或几十个候选，追求覆盖率、召回率和候选质量；排序或重排层负责在小候选集上做更细的打分和组合优化，追求点击、满意度、多样性、长期价值。项目里基础内容推荐和协同过滤提供传统 baseline，`Word2Vec/GloVe` 召回补充语义和序列共现信息，最后 `PPO` 在候选集上做重排。

- `多路召回之间是相加还是去重拼接`：先做去重拼接，再按业务规则或打分融合进入后续重排。

- `为什么不用 RL 直接做全链路`：核心答案是工程可行性和样本效率。全量电影库下动作空间可能上千上万，用户真实反馈又很稀疏，on-policy 的 `PPO` 很难稳定学到有效策略。而重排阶段候选集已经被召回层缩小，比如项目里取 `top_n * 3` 的候选池，这时策略网络只需要在更小、更高质量的集合里决定展示顺序。所以把 RL 放在最有优势的位置。

- `为什么不用监督学习直接排序`：监督排序更擅长点式相关性预测，RL 更适合把列表级奖励和长期反馈纳入目标。

- `候选集怎么构建`：我不是直接用一个单点算法召回，而是先做多路候选。文本侧用电影标题、类型、导演、国家、简介生成 `word2vec` 和 `glove` 风格的语义向量，交互侧把用户正反馈序列当作 item sequence，训练 `item2vec / 共现式嵌入`。然后把这几路分数融合成 `semantic_hybrid`。如果用户无历史、召回不足或语义路结果太少，就降级到基础内容推荐、协同过滤和热门片补齐。

- `为什么候选池设成 top_n * 3`：这是精度和计算量之间的折中，太小会限制重排空间，太大会增加计算和噪声。

- `semantic_hybrid 的核心思想`：把“电影文本相似”和“用户交互共现”两种语义信号合在一起，避免只看文本或只看行为导致的偏差，本质是“内容理解 + 行为共现”的混合召回。

- `两路分数怎么融合`：做加权融合，权重由离线实验调，项目里也是按多路得分组合后再排序。

- `为什么选 Word2Vec/GloVe，不直接上 BERT`：`Word2Vec/GloVe` 训练轻、部署轻。先用轻量语义向量把语义短板补齐，等数据量和算力上来后，再把 `BERT` 作为离线预编码特征接到召回或排序里。

- `用户状态向量在 PPO 重排里怎么定义`：先把用户历史转换成状态向量，里面会包含用户喜欢过的电影、评分倾向、行为强度，以及由电影特征和语义标签聚合出来的偏好表征。这样做有两个目的：第一，降低原始行为序列的噪声；第二，把策略学习问题转成“给定用户状态，如何在候选集上做更优排序”。

- `动作空间怎么定义`：动作空间不是整个电影全集，否则样本极稀疏。先由召回层提供候选电影，再由 `PPO` 根据用户状态和候选特征给每个动作分配概率，进而决定重排顺序。代码里会先构建电影特征矩阵，再用策略网络输出 `policy_score`，之后叠加 `coverage bonus` 和 `diversity bonus` 形成最终的 `rl_score`。

- `动作是选一个 item 还是选一个列表`：项目实现上更接近逐步选 item 的列表构造，但目标是优化最终列表效果。

- `奖励函数怎么设计`：主奖励来自行为类型和评分强弱，比如高评分是更强的正反馈，收藏和点击给较弱正反馈。然后额外设计两个辅助项：一个是覆盖率奖励，曝光少的电影会拿到更高 bonus，避免系统长期只推热门头部；另一个是多样性奖励，如果候选电影和已经选出的片子太像，它的 bonus 会下降。辅助奖励权重过大时会伤害相关性，所以最终权重要靠离线实验校准。

- `为什么重排里加入 ε-greedy`：纯 exploitation 会让系统越来越保守，`ε-greedy` 能给长尾和新物品少量探索机会。只按当前最高分推荐，看起来短期指标可能不错，但系统会越来越偏头部和既有兴趣，导致新片和长尾片没有曝光，也拿不到训练数据。项目里在重排阶段加入 `ε-greedy`，当触发探索时，会按覆盖率相关权重随机挑选候选，而不是永远选最高 `rl_score` 的电影。

- `覆盖率奖励和多样性奖励为什么都要加`：覆盖率解决“总推热门”，多样性解决“推得很像”。`coverage bonus` 通过曝光次数反向加权，`diversity bonus` 通过候选特征和已选电影特征的相似度计算。两个奖励一起加，能让推荐结果既不过度头部化，也不会内容同质化。

- `模型版本管理`：项目训练完成后，会把 `PPO` 权重保存到本地目录，再把版本号、训练时间、关键指标等写入数据库。后台能展示版本列表，支持当前激活版本查看、历史版本回滚和旧版本清理。

- `离线评估怎么做`：先从用户历史评分里取正反馈样本，比如高分电影；然后对每个用户做 `leave-one-out`，把最后一个正样本作为测试目标，剩下行为构成训练偏好。接着用不同算法给出 top-k 推荐，看目标电影是否出现在结果里，并计算 `precision@k`、`recall@k`、`ndcg@k`。这能比较基础热门、基础内容、基础协同过滤、语义混合以及 `PPO` 重排之间的差异。局限性是离线指标不能完全代表线上收益。

- `为什么用 NDCG，不只看 Recall@K`：因为 `NDCG` 会考虑命中位置，越靠前命中价值越大，更接近真实排序质量。

- `为什么用留一验证，不是随机切分`：推荐数据有时间性，留一验证更符合“用过去预测未来”的实际场景。

- `怎么处理冷启动`：用户冷启动和物品冷启动分开看。用户冷启动走热门片和内容语义，物品冷启动优先依赖电影文本特征。用户没有历史行为时，协同过滤和交互嵌入基本失效，这时就回退到热门电影和内容语义召回，比如根据电影类型、导演、简介文本做相似推荐。对于新电影，由于还没有足够曝光和评分，行为特征也不足，所以主要依赖结构化信息生成特征，让它有机会进入候选池。

- `为什么说这个项目有效`：三层证据。第一层是定量证据，用离线指标对比 baseline、语义混合和 `PPO` 重排；第二层是实验过程证据，后台实验页能展示多次快照、不同算法曲线和模型版本；第三层是机制解释，语义召回能改善冷启动，`PPO` 重排能改善多样性和覆盖率。

- `如果离线指标提升不明显怎么办`：优先从数据量、特征表达和奖励设计三方面分析。

- `项目最容易被质疑什么`：数据规模小、RL 是否必要、单机架构是否太轻、为什么不用工业大模型和分布式架构。我的回答是：把它定位成“可运行的推荐闭环原型”，重点展示方法和可演进性。RL 是验证策略优化思路的增强层，不是强行替代整个推荐系统。

- `升级到工业级怎么演进`：按特征、召回、排序、实验平台四层演进。引入更稳定的序列模型做召回，排序侧上 `Wide&Deep / DeepFM / DIN / DCN / MMoE` 这类工业常见模型，RL 保留在重排或流量分配层；再引入更完整的实验系统，比如 `AB` 测试、特征版本管理、实时监控和告警。

- `RL 最适合放在哪`：通常更适合放在重排、流量调控、长期价值优化这些策略层。

- `为什么这是推荐闭环`：它包含数据采集、行为回流、模型训练、版本管理、实验对比和评估，形成了从反馈到优化的闭环。因为用户行为会继续写回数据库，经验会入池，达到阈值后触发训练，训练产出新模型。同时它覆盖了经典推荐、语义建模、强化学习重排、离线评估。

- `为什么没有一开始就上双塔、DIN、BERT4Rec`：一开始就上双塔、DIN 或 BERT4Rec，工程复杂度会明显升高，效果却不一定稳定。小项目最需要的是可对比的 baseline 和清晰的优化路径。后续如果做工业级演进，我会优先考虑双塔召回和序列排序，优先升级召回，用双塔或序列召回提升候选质量。

## 推荐系统与机器学习题

- `什么是过拟合，怎么处理`：过拟合通常表现为训练集指标很好，但验证集和测试集明显变差。推荐场景里更容易过拟合，因为行为稀疏、特征维度高、长尾严重，而且一些特征高度 id 化。处理方法通常有几类：模型层面用 `L1/L2` 正则、dropout、减少参数量；数据层面用更合理的负采样、按时间切分验证集、去掉泄露特征；特征层面限制稀疏 id 直接记忆，加入统计特征和共享 embedding。

- `L1 和 L2 正则`：`L1` 在目标函数里加权重绝对值和，优化后很多参数会被压成 0，所以有天然的特征选择效果；`L2` 加的是平方和，会更平滑地惩罚大权重，让模型不至于过度依赖少数特征。实际工程里深度学习模型更常用 `L2` 或 `weight decay`，因为优化更稳定；线性模型和高维稀疏特征场景里，`L1` 更有吸引力。

- `BatchNorm 和 LayerNorm`：`BatchNorm` 沿 batch 维做归一化，适合 `CNN`；`LayerNorm` 沿特征维做归一化，更适合 `Transformer` 和序列建模。`BatchNorm` 依赖同一个 mini-batch 里的样本统计量，在 batch 较大、分布相对稳定的视觉任务里很有效；但在序列模型、在线推理和 batch 很小时，它容易不稳定。`LayerNorm` 对单个样本内部的隐藏维做归一化，不依赖 batch 大小，适合 `Transformer`，所以 `LLM` 几乎都用 `LayerNorm/RMSNorm` 而不是 `BatchNorm`。

- `Dropout 的作用，为什么推理时不用`：核心思路是训练时随机把一些神经元置零，让模型不能过度依赖某些局部通路，等价于对很多子网络做近似集成。这对小数据和容易过拟合的场景尤其有效。推理阶段如果还随机屏蔽神经元，输出会不稳定。推荐场景里，dropout 常用于 `MLP` 排序模型或 embedding 后的深层网络。本质是正则化，而不是提升表达能力。

- `为什么分类常用交叉熵，不用 MSE`：交叉熵衡量的是预测分布和真实分布之间的差异，天然对应最大似然估计，因此非常适合分类问题，特别适合 `softmax` 或 `sigmoid` 输出；`MSE` 在分类里容易带来梯度问题，更适合回归，因为它把标签当连续值处理。推荐系统里点击率建模、曝光点击预测、转化预测一般默认就是 `logloss`，也就是交叉熵损失。评分是连续值或近似连续值，本质更接近回归问题。

- `AUC、GAUC 和 LogLoss`：`AUC` 是随机抽一个正样本和一个负样本，模型把正样本排在前面的概率，更关注排序而不是绝对概率；`GAUC` 是按用户或组加权后的 `AUC`，是在用户、query 或流量组内先算 `AUC`，再按曝光量加权平均，所以在推荐和广告更常见；`LogLoss` 对预测概率是否靠谱非常敏感，看概率校准和整体拟合程度。

- `为什么推荐常看 GAUC`：因为用户分布不均衡，直接算整体 `AUC` 可能被头部用户或头部流量主导。

- `Precision@K、Recall@K、F1、NDCG@K`：`Precision@K` 是推荐列表里有多少是真正相关的，强调前几条够不够准；`Recall@K` 是用户实际喜欢的内容有多少被找回来了，强调不要漏太多；`F1` 是 `precision` 和 `recall` 的调和平均，更适合综合平衡；`NDCG@K` 则考虑命中的位置，越靠前命中价值越大，所以更接近用户真实浏览行为。

- `样本不平衡怎么处理`：推荐和广告数据天然极不平衡。常见做法有重采样、loss 加权、阈值调整、难例挖掘和更合理的评估指标。具体可以做负样本下采样、正样本过采样、`focal loss` 或 `class weight`；还要注意线上推断时分布偏移。

- `负采样`：这是推荐面经高频题。常见有随机负采样、按曝光日志采样、按时间窗口采样和难负样本采样。优先从曝光未点击里采负样本，因为这些才是真正用户“看到了但没选”。负采样的核心不是数量，而是尽量贴近真实决策边界。

- `Embedding 的本质`：把高维稀疏离散特征映射成低维稠密向量，既降维，又能学习语义相似性。离散 id，比如用户 id、电影 id、导演、类型、关键词，embedding 把每个离散 token 映射成一个可训练向量，承载统计共现和语义关系。

- `连续特征和离散特征怎么处理`：离散特征常见做法是 one-hot 后 embedding 或直接哈希到 embedding 表；连续特征像评分均值、热度、时长、曝光次数，通常需要标准化、归一化或分桶。比如树模型更喜欢离散分桶后的特征，深度模型则可以直接接连续值。是否要做 `log` 化、分桶化，取决于分布是否长尾。

- `SGD 和 Adam`：`SGD` 只有统一学习率，更新方向更朴素，有时收敛慢，但泛化性能常不错；`Adam` 结合了一阶矩和二阶矩估计，相当于对不同参数使用不同步长，所以在稀疏特征、复杂深网和前期训练阶段更常见。由于 embedding 大量存在、特征稀疏明显，`Adam` 往往更常用，本质是收敛速度和泛化特性的权衡。

- `学习率`：学习率决定每次参数更新步子有多大，太大不收敛，太小收敛慢；常见策略有 `warmup`、`step decay`、`cosine decay` 和自适应优化器。

- `warmup`：尤其常见于 `Transformer` 和大模型，让模型在训练初期平稳进入有效更新区间，避免大步长导致不稳定。因为刚开始参数还不稳定，直接用大学习率容易炸。对推荐系统来说，embedding 和 `MLP` 往往也会用不同学习率，它是最值得优先调的超参之一。

- `梯度消失和梯度爆炸`：梯度消失是反向传播时梯度越来越小，前层几乎学不到；梯度爆炸是梯度过大导致参数更新失控。前者会让模型学不动，后者会导致损失发散、数值溢出。梯度裁剪主要解决后者。

- `常见激活函数`：`Sigmoid` 输出在 0 到 1，适合做概率，但在两端容易饱和，梯度变小；`Tanh` 零中心一点，但同样有饱和问题；`ReLU` 计算简单、梯度稳定，是经典默认选择；`GELU` 可以看成更平滑的门控激活。隐藏层一般倾向 `ReLU` 或 `GELU`，输出层再根据任务选 `sigmoid`、`softmax` 或线性层。`ReLU` 可能出现 `dying ReLU`。

- `Softmax`：对 logits 做指数化再归一化，把一组实数映射成概率分布，值越大对应概率越高，且所有概率和为 1。推荐里 `softmax` 也常用于多候选打分归一化、注意力权重计算和策略网络动作概率输出。`PPO` 重排里用到了类似思想，把每个候选电影的打分变成概率分布。

- `Attention 为什么除以 sqrt(d_k)`：做尺度归一化，为了稳定；否则 `softmax` 饱和，梯度变差。

- `Transformer Encoder 的核心结构`：多头自注意力 + 前馈网络 + 残差连接 + LayerNorm。从不同子空间关注不同关系，比如语义、位置或行为模式。推荐里的 `SASRec`、`BERT4Rec` 以及大模型本身都建立在这套结构上。

- `残差连接的作用`：帮助信息和梯度穿透深层网络，缓解训练困难。

- `特征交叉`：显式或隐式建模“特征和特征一起出现”。传统做法会手工构造交叉特征，深度模型则通过 `FM、DeepFM、DCN、MLP` 等自动学习交互。推荐特别依赖交叉，因为用户偏好常常不是线性的，特征之间存在强耦合。

- `为什么推荐要分召回、粗排、精排、重排`：召回层面向海量物品，目标是快速筛出一小批潜在相关候选，所以更看重速度和覆盖；粗排和精排面向越来越小的候选集，模型会更复杂、更注重点击率和转化率；重排则通常不再只追求单点相关性，而是考虑多样性、新颖性、内容生态、长期收益等列表级目标。

- `多路召回怎么融合`：常见做法有去重拼接、分数归一化后加权融合、规则保底和学习式融合，核心是兼顾覆盖和相关性。热门召回偏头部，协同过滤偏行为共现，内容召回偏语义相似，双塔召回偏向学习到的向量邻近。项目里 `semantic_hybrid + baseline_hybrid + hot` 的回退思路，本质就是规则式融合。

- `UserCF 和 ItemCF`：`UserCF` 找相似用户，`ItemCF` 找相似物品；前者更依赖用户重叠，后者通常更稳定、更适合商品和内容推荐。

- `矩阵分解`：把用户-物品交互矩阵分成用户向量和物品向量的乘积，把“相似性”转成潜在空间里的内积关系，把用户和物品映射到共享隐空间里，能在稀疏场景下学到更稳的潜在偏好，比纯相似度法更有泛化能力。常见方法包括 `SVD、ALS、BPR`。优势是泛化更强、可扩展性更好；缺点是对冷启动和文本语义不够友好。处理隐式反馈时，常见做法是把点击、收藏转成偏好强度，并配合置信度或 `pairwise loss`。

- `双塔召回`：用用户塔编码用户兴趣，用物品塔编码 item 特征，训练目标通常是让正样本用户-物品对更接近、负样本更远。训练好后，物品向量可以离线预计算并建索引，线上只需算用户向量，再做近邻检索。缺点是用户和物品在最终打分前交互不充分，所以精细建模能力不如交叉特征很强的排序模型。

- `Wide&Deep`：推荐系统经典模型。Wide 侧通常是线性模型加人工交叉特征，擅长记住历史上反复出现的强规则；Deep 侧则通过 embedding 和多层感知机学习更高阶、更平滑的泛化能力。它兼顾“记忆”和“泛化”。

- `DeepFM`：用 `FM` 自动学习低阶交叉，擅长捕捉二阶特征交互，不需要人工手写很多交叉；再用 `DNN` 学高阶交叉，减少了 `Wide&Deep` 对人工特征工程的依赖。`FM` 适合稀疏特征，因为它通过向量内积表示交互，比手工枚举所有交叉更高效。

- `DCN`：本质是把显式特征交叉做成网络层。相较于纯 `MLP` 依赖隐式学习交互，`Cross Network` 能更直接地构造有限阶组合特征。`Wide&Deep、DeepFM、DCN` 可以看成显式交叉和隐式交叉的不同路线。

- `DIN`：传统做法会把用户历史行为平均池化成一个固定向量，但这会把“用户喜欢什么”平均掉。`DIN` 用 attention 让候选 item 和历史行为做交互，得到一个 `target-aware` 用户表示，解决把不同兴趣混成一个固定向量、缺少候选感知的问题。核心是“兴趣不是静态的”。

- `DIEN 和 SIM`：`DIEN` 在此基础上引入兴趣抽取层和兴趣演化层，会用 `GRU` 一类结构建模行为序列，强调“用户兴趣不是静止向量，而是随时间变化的状态”。`SIM` 更关注超长行为序列的高效检索和匹配。

- `曝光未点击能直接当负样本吗`：曝光未点击比“没见过”更接近真实负反馈。

- `推荐里的偏差`：曝光偏差是某些内容从来没机会被看见；位置偏差是越靠前越容易被点；流行度偏差是热门内容因为已有反馈更多而更容易继续被推；反馈回路偏差是系统推荐什么，就更容易收集到什么数据。这些偏差会让模型把“被展示”误当成“天然更好”。项目里通过覆盖率奖励和探索机制，实际上就在缓解部分偏差积累。

- `树模型和深度模型怎么选`：`XGBoost/LightGBM` 这类树模型在结构化表格数据上很强，调起来相对稳，特征重要性也更容易解释；但它对大规模稀疏 id、embedding 联合学习、多模态输入和序列行为建模没那么自然。深度模型虽然调参更复杂，但更适合用户 id、item id、文本向量、序列行为一起建模。

## 强化学习扩展题

- `强化学习的四要素`：状态、动作、奖励、环境。在环境中根据状态选择动作，收到奖励后继续更新策略。推荐场景里，状态可以是用户画像和上下文，动作可以是推荐哪个 item 或列表，奖励可以是点击、观看、停留、购买等，环境就是用户与系统的交互过程。

- `把推荐建成 MDP`：把用户当前画像和上下文当状态，把推荐决策当动作，把用户反馈当奖励，把用户行为变化看成状态转移，就可以近似建成 `MDP`。

- `MAB`：`Multi-Armed Bandit` 是强化学习中经典的探索与利用问题。每次面对一个上下文，选择一个动作，立即获得奖励，但默认当前动作不会影响未来环境结构，所以没有复杂的状态转移问题。这适合新闻推荐、流量分配、探索利用等场景。当决策周期短、未来影响弱或建模成本太高时，bandit 更实用；当明确关心长期留存和序列效应时，才更有必要上 RL。

- `On-policy 和 Off-policy`：`On-policy` 的代表是 `PPO`，用当前策略生成的数据学当前策略，所以训练稳定但样本利用率相对低；`Off-policy` 的代表是 `DQN` 一类，可以用其他策略收集的数据学习目标策略，因此样本效率更高，但训练容易不稳定。小项目里，`PPO` 这类 on-policy 或近 on-policy 方法因为实现简单、稳定性更好。

- `Q-learning 和 DQN`：`Q-learning` 的核心思想是学习状态-动作价值 `Q(s,a)`，先学价值再选动作，每次选价值最大的动作。`DQN` 就是用深度网络来近似 `Q` 函数，同时引入经验回放和 target network 提升训练稳定性。推荐系统里不用太多 `DQN`，因为动作空间经常是动态候选集、列表级动作，而且点值函数方法不一定最适合序列重排，所以很多场景更偏向策略梯度或 `actor-critic`。

- `经验回放解决什么问题`：打破样本相关性，提高数据利用率和训练稳定性。

- `策略梯度`：直接对参数化策略求梯度，目标是提高高回报动作的概率、降低低回报动作的概率。

- `Actor-Critic`：Actor 负责出动作策略，Critic 估计价值函数或优势函数，既能保持策略优化的灵活性，又能减小更新噪声。Critic 不负责最终执行动作，而是提供学习信号。

- `PPO 的核心思想`：继承了策略梯度和 `actor-critic` 的思想，重点解决“策略一下子更新太猛，训练容易崩”的问题。`PPO` 会比较新旧策略在同一动作上的概率比值，如果变化过大，就通过 `clip` 机制限制目标函数收益，防止策略一步跨太远。这样做不像 `TRPO` 那样需要复杂约束优化，实现更简单，却能保持不错的训练稳定性。`PPO` 做重排很合理：小规模策略优化上足够轻量，也方便解释。

- `PPO 为什么比原始 policy gradient 更稳`：因为它限制了更新步长，避免新策略和旧策略差异过大。

- `PPO clip objective 在做什么`：限制策略漂移。目标函数里会出现新旧策略概率比值 `ratio`。如果某个动作回报很好，模型本来会想大幅提高它的概率，这种大幅更新会让训练不稳定。所以 `PPO` 用 `clip` 把 `ratio` 限制在一个区间，比如 `[1-eps, 1+eps]`。项目里对应的是 `PPO_CLIP_EPSILON`。

- `GAE 和 advantage`：`GAE` 是 `Generalized Advantage Estimation`，是一种把多步 `TD` 误差加权累积起来的优势估计方法，可以在偏差和方差之间做平衡。为什么要估计 advantage：因为 `A(s,a)=Q(s,a)-V(s)`，它表示在当前状态下，这个动作比平均水平到底好多少。

- `为什么 PPO 适合做推荐重排`：推荐重排不像召回那样面对海量物品，而是在有限候选里做顺序决策。这时 RL 的两个优势更明显：第一，可以把覆盖率、多样性、探索等非单点相关性目标写进奖励；第二，可以考虑短期满意度和长期反馈之间的权衡。`PPO` 比很多 RL 算法实现更轻、更稳，所以适合作为项目级重排方案。

- `探索与利用`：推荐系统需要在探索和利用之间找平衡。经典方法包括 `ε-greedy`、`UCB`、`Thompson Sampling`、bandit 策略和 RL 策略约束。项目在 `PPO` 重排里用小比例 `ε-greedy`，就是一种轻量探索机制。

- `推荐系统为什么天然过度利用`：因为已有高反馈内容会持续拿到更多曝光，形成自增强闭环。

- `PPO 和 ε-greedy 怎么配合`：`PPO` 会先根据用户状态和候选电影特征算出策略分数，再叠加 `coverage bonus` 得到初始 `rl_score`。随后在构造最终列表时，大部分情况下按综合分数选最优候选；少部分情况下触发 `ε-greedy`，按覆盖率相关权重随机选。这样给 `PPO` 一个受控探索出口。

- `为什么探索时按覆盖率加权，不是完全均匀随机`：因为希望更有针对性地补偿低曝光内容，而不是无差别随机。

- `Reward shaping`：在主奖励外加入辅助奖励帮助学习，但如果设计不当，模型会“钻奖励漏洞”而偏离真实目标。它的价值是让训练更稳定、目标更可学；风险是模型可能学会优化代理目标而不是真实目标。

- `离线 RL`：看起来很适合推荐，因为线上探索成本高，大家都想直接用历史日志学策略。但问题是，历史日志只覆盖了旧策略实际做过的动作。如果新策略想选一些日志中几乎没出现过的动作，模型就必须对“分布外动作”做外推，这非常不可靠。于是 `Q` 值容易被高估，策略会学到看似很优但真实表现很差的行为。

- `PPO、DPO、GRPO`：`PPO` 来自传统强化学习，依赖策略梯度和价值估计，强调稳定更新；`DPO` 主要用于偏好对齐，它不显式训练 reward model 和 value model，而是直接用偏好对比较优化策略，更像“有偏好标签的对数比率学习”；`GRPO` 则是一些大模型后训练里使用的组相对优化思路，用样本组内相对表现来更新策略。你的项目采用 `PPO`，因为这里问题是推荐重排策略优化，不是大模型人类偏好对齐。

- `RL 不一定比监督学习排序更好`：监督学习排序直接用大量点击日志学当前相关性，训练稳定、样本多、评估也成熟；RL 只有在明确关心长期价值、探索利用、列表级约束或多步决策时，优势才会更明显。而且 RL 对环境建模、奖励设计和分布偏移都很敏感。

## 大模型扩展题

- `Embedding 模型和 Reranker`：Embedding 模型把文本、用户描述或 item 描述编码成向量，适合用 `ANN` 做大规模近邻检索，所以特别适合召回。Reranker 通常会对 query 和候选做更充分的交互式编码，比如 `cross-encoder`，因此相关性判断更准确，但计算成本高，不适合全量检索。

- `大模型在推荐里常见的三种用法`：第一做文本理解和特征增强，第二做排序或重排辅助，第三做对话式推荐和推荐解释。

- `大模型更适合做召回、排序还是重排`：现阶段更适合小候选集的重排、解释和冷启动增强，不太适合大规模主召回。主召回最强调吞吐和延迟，更适合双塔、向量检索和轻量召回模型；主排序也要扛较大流量，深度 `CTR/排序` 模型更划算。大模型通常适合小候选集重排、高价值 query、复杂语义理解、理由生成和对话交互。

- `如果把大模型接进当前项目`：不会让大模型直接替代现有推荐主链路，而是作为增强模块接入。第一步，用 `LLM` 离线解析电影简介、评论和标签，生成更细粒度语义标签，提升内容理解；第二步，在 `semantic_hybrid` 召回出的少量候选上，用 `LLM` 做辅助重排，特别针对冷启动和长尾语义匹配；第三步，在最终结果展示层生成推荐理由。
