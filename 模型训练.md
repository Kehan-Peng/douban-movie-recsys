# 模型训练

这份文档专门面向“算法实习生岗位”的细粒度训练拷打，默认从你的项目“基于强化学习的 Python 豆瓣电影推荐系统”出发来回答。

有两个前提先说清：

- 当前项目的真实训练链路，核心是 `Word2Vec-style / GloVe-style` 本地语义向量训练 + 本地 `PPO` 重排训练，主要实现基于 `NumPy`，不是 `PyTorch` 大模型微调项目。
- 但很多面试官会从你的项目一路追问到 `SFT / Reward Model / LoRA / RAG / FAISS / PyTorch`。这些问题即使项目没直接用，也要会答，而且要答成“如果我把当前项目扩展到工业级，我会怎么做”。

# 训练目标与样本设计

## 为什么这个项目做强化学习，不做 SFT？
短答：因为这个项目优化的是“推荐列表决策”和“用户反馈奖励”，不是生成式文本模仿。`SFT` 更适合学“标准答案长什么样”，`RL` 更适合学“一个动作带来的长期收益”。

展开：如果面试官这么问，核心要说清任务类型。`SFT` 需要相对稳定的监督标签，比如“这个输入对应什么标准输出”。但电影推荐重排里，系统面对的是候选集、用户状态和行为反馈，目标不是模仿一个固定排序答案，而是根据点击、收藏、评分、多样性、覆盖率这些反馈动态调整策略。当前项目里的 `PPO` 就是拿用户状态向量和候选电影特征做策略优化，属于典型的决策问题。  
如果把项目换成 LLM 推荐解释生成，那一层可以先做 `SFT`；但你现在这个项目的核心不是生成文本，而是优化推荐动作，所以优先是 `RL` 而不是 `SFT`。

## 为什么不用“监督学习排序”直接替代 PPO？
短答：监督排序能很好学“相关性”，但很难直接优化“多样性、覆盖率、探索、长期反馈”这类列表级目标，所以我把 PPO 放在重排层。

展开：如果业务目标只是短期点击，监督学习排序通常更稳、更省事。但你的项目里想让推荐结果不只“像”，还要“别全是热门”“别全是同质化电影”，并且希望通过探索收集更多真实反馈。这些目标很难都压成一个静态标签。于是当前项目做法是先用传统和语义召回拿到候选，再让 PPO 对候选做策略化重排，把相关性、多样性、覆盖率一起纳进 reward。这样讲，面试官通常会认可你的选型是“分层使用 RL”，而不是滥用 RL。

## 你的训练样本粒度是什么？
短答：当前项目主要有两种粒度：语义召回训练是“token/序列粒度”，PPO 训练是“用户状态-电影动作-反馈奖励”的交互粒度。

展开：这题很细，但经常能区分会不会做项目。  
`Word2Vec/GloVe` 文本侧的样本粒度，是电影文本 token 序列；交互侧的样本粒度，是用户正反馈形成的电影序列，比如 `movie_12 -> movie_45 -> movie_78`。  
`PPO` 训练样本则是更标准的 RL 粒度：状态 `state`、动作 `movie_id`、旧策略概率 `old_prob`、奖励 `reward`、辅助奖励 `aux_reward`。  
如果面试官继续追问“样本粒度和特征粒度的区别”，你可以答：样本粒度说的是一条训练记录的最小单位，特征粒度说的是一条记录里到底包含用户级、物品级、上下文级还是列表级信息。

## 你的特征粒度怎么分？
短答：我把特征大致分成用户粒度、电影粒度、用户-电影交叉粒度，以及列表粒度的辅助信号。

展开：  
用户粒度包括历史高分电影、行为强度、偏好画像。  
电影粒度包括题材、导演、国家、简介文本、语义向量。  
用户-电影交叉粒度在当前项目里没有上复杂深度交叉网络，但通过“用户历史加权 profile 和候选向量相似度”间接体现了交互。  
列表粒度则主要出现在 PPO 重排阶段，比如当前列表里已经选了什么、候选与已选内容有多相似、当前电影曝光是否过多。  
面试里说到这一步，基本就不像背八股了，而是真的理解训练对象。

## 你的正样本和负样本怎么定义？
短答：当前项目里显式高评分、收藏、点击视为不同强度的正反馈；没有做严格的曝光级负采样，更多是通过离线排序竞争和 RL 反馈强弱来区分。

展开：这题一定要诚实。你的项目不是大规模 CTR 预估系统，没有完整曝光日志，所以不能硬说“我有严格曝光负样本体系”。更准确的说法是：  
对语义召回训练，序列里的正共现来自用户的正向行为序列；  
对 RL 奖励，评分、收藏、点击分别映射成不同强度奖励；  
没有发生行为的物品，并不被简单粗暴地全部当真负样本。  
如果面试官顺势问工业界怎么做，你就答：推荐系统里更偏向用“曝光未点击”做相对可信的负样本，而不是把未曝光物品全判负。

## 你的 reward 是怎么从业务行为映射出来的？
短答：当前项目里高评分是强正反馈，收藏是中等正反馈，点击是弱正反馈；此外还叠加覆盖率奖励。

展开：项目代码里已经把这件事写得很明确：  
评分行为 `behavior_type == 1` 时，用 `(score - 5) / 5` 做归一化奖励，也就是 5 分附近近似 0，越高越正；  
收藏给固定正奖励 `0.35`；  
点击给更弱的正奖励 `0.15`。  
另外还会加上覆盖率辅助奖励，也就是曝光越少的电影 bonus 越高。  
这里的答法重点不是公式多复杂，而是让面试官知道你理解“强弱反馈分层”和“代理目标”。

## 为什么点击奖励比评分奖励小？
短答：因为点击是弱反馈，噪声大，很多时候只是封面或标题吸引；评分更接近真实满意度。

展开：推荐系统里不同反馈信号价值差异很大。点击说明“用户愿意点开”，但不代表真喜欢；收藏比点击更强，因为有延迟决策含义；评分最接近显式满意度，所以在当前项目里被赋予更大的可变奖励。这个回答是很标准的“反馈分层建模”思路，算法岗面试里非常常见。

## 为什么要有 `aux_reward`，不能只看主奖励？
短答：因为主奖励太稀疏，只靠主奖励很难把“长尾曝光”和“列表体验”学进去，所以需要辅助奖励稳定训练。

展开：你的项目主奖励主要来自用户行为，但真实高质量反馈并不密集。如果只靠主奖励，策略容易一味偏向头部高反馈电影，导致长尾完全学不到。`aux_reward` 在当前项目里更多承担“把业务偏好显式注入策略”的作用，比如 coverage bonus。  
回答这题时，一定要补一句：辅助奖励不能过大，否则策略会学偏，变成为了“覆盖率好看”而牺牲推荐相关性。

## 为什么经验池要攒到一定量再训练？
短答：因为单条反馈太噪，逐条更新会让策略震荡，批量训练更稳定。

展开：这道题的关键在“稳定性”三个字。推荐日志天然 noisy，用户可能误点，也可能今天喜欢、明天不喜欢。PPO 如果每来一条就更新，策略会特别容易过拟合近期噪声。你的项目做法是把经验先写入池子，达到阈值后再批量训练。这个回答既符合代码实现，也符合推荐系统里的近线训练思路。

# 召回、候选与检索细节

## 你的业务里有哪些召回通道？
短答：当前项目可讲成 7 路：`hot`、`baseline_content`、`baseline_cf`、`baseline_hybrid`、`word2vec_content`、`glove_content`、`word2vec_cf`、`glove_cf`，对外主候选融合成 `semantic_hybrid`，再给 PPO 重排。

展开：更准确地说，主候选来源是语义增强召回，也就是文本侧 `word2vec/glove` 和交互侧 `word2vec/glove` 的融合；当结果不足时，回退到基础内容推荐、基础协同过滤和热门补齐。  
如果面试官问“到底几路召回”，你不要机械回答“8 路”，而是说：逻辑上是“热门兜底 + 传统 baseline + 4 路语义通道 + 1 路融合通道”。这样更像真正理解系统结构。

## 每路召回的量级怎么定？
短答：当前项目是小规模单机场景，最终候选池一般取 `top_n * 3`；如果按工业口径回答，我会把每路召回做成几十到几百，再统一融合。

展开：面试官有时候会故意问你“召回量级”。你项目代码里 `recommend_movies` 是先拿 `candidate_pool_size = max(top_n * 3, top_n)`，也就是如果最终展示 10 条，候选池默认大约 30 条。这在小项目里很合理，因为总物品量本来不大。  
但如果面试官想听工业实践，你可以补：真实业务里每路召回可能是 `50~500`，再统一合并、去重、打散后送给粗排或精排。

## 为什么候选池不是越大越好？
短答：候选太小会限制重排空间，太大又会引入噪声和额外计算，所以要在覆盖和精度之间平衡。

展开：候选池大小是很实战的问题。太小时，PPO 重排几乎没有发挥空间；太大时，很多低质量候选会污染排序，反而让重排更难学。你的项目用 `top_n * 3` 是一个保守但自洽的配置，适合当前数据规模。面试时强调“这是覆盖率和计算量之间的折中”，就很稳。

## 你的检索是怎么做的？
短答：当前项目主要是本地向量/分数计算 + 精确排序，还没有引入真正的 ANN 检索。

展开：这题要非常坦诚。你现在这个系统没有 FAISS，也没有 IVF/HNSW 之类索引；文本和交互语义召回主要还是基于本地 embedding map 做相似度计算和排序。这在单机、小物品库里完全可以接受。  
如果面试官问“那你了解 ANN 吗”，就顺着答：当前项目还没到必须上 ANN 的规模，但如果电影库继续扩展，我会优先考虑 FAISS。

## ANN 是什么？你了解哪些方法？
短答：ANN 是 Approximate Nearest Neighbor，核心是牺牲少量精度换取大幅检索加速；常见方法有 `Flat`、`IVF`、`PQ`、`IVF-PQ`、`HNSW`。

展开：  
`IndexFlatL2 / IndexFlatIP` 是精确搜索，不是近似，但常用作 baseline。  
`IVF` 先做粗聚类，再只搜少量倒排桶。  
`PQ` 把向量压缩成短码，节省内存。  
`IVF-PQ` 是工业里很常见的速度/内存折中方案。  
`HNSW` 是图索引路线，检索速度和召回率通常都不错，调参也相对直观。  
如果面试官问“什么时候用哪个”，你就答：小库先 `Flat`，中大库看时延和内存约束，在 `IVF`、`IVF-PQ`、`HNSW` 里选。

## FAISS 具体有哪些常见索引？
短答：最常见的是 `IndexFlatL2/IP`、`IndexIVFFlat`、`IndexIVFPQ`、`IndexHNSWFlat`。

展开：  
`Flat` 最准，但慢。  
`IVFFlat` 先聚类、后桶内精查，适合中等规模。  
`IVFPQ` 进一步压缩，适合更大规模、内存敏感场景。  
`HNSWFlat` 适合高召回、低时延的图搜索。  
如果面试官继续问参数：`nlist` 控制聚类桶数，`nprobe` 控制搜索桶数，`m / nbits` 是 PQ 压缩参数，`efConstruction / efSearch` 是 HNSW 的建图和检索强度参数。

## 当前项目如果接 FAISS，你会怎么接？
短答：我会先把电影向量离线编码成 `float32`，用 `IndexFlatIP` 做 baseline，再根据规模换到 `IVF` 或 `HNSW`。

展开：你项目里最容易接 FAISS 的地方是语义召回。流程会是：  
先把电影文本向量或交互向量离线算好；  
统一归一化后存成 `float32`；  
用内积或余弦对应的索引建立库；  
线上用户向量进来后，先 ANN 召回 top-k，再走现在已有的重排逻辑。  
这样答法既承认当前没做，也说明你知道怎么扩。

## 召回模型和精排模型在特征侧最大的区别是什么？
短答：召回特征偏粗、偏稳、偏可预计算；精排特征偏细、偏交互、偏上下文。

展开：这是推荐系统非常经典的细节题。  
召回侧最重要的是快，所以更倾向用户/物品主干 embedding、长期偏好特征、稳定标签、类目、文本向量等能提前算好的特征。  
精排侧因为候选少得多，可以加更多细特征，比如上下文、会话特征、用户-电影交叉特征、序列特征、统计特征、甚至列表位置和曝光特征。  
你项目里其实就是这样：语义召回主要看 embedding，相对粗；PPO 重排时开始考虑 coverage、多样性和当前已选列表。

## 冷启动怎么处理？
短答：用户冷启动靠热门片和内容语义，电影冷启动靠文本元信息和语义向量，不完全依赖行为数据。

展开：  
新用户没有历史时，`CF` 和交互序列 embedding 几乎帮不上忙，所以主要依赖热门片和内容相似度。  
新电影没有行为时，协同信号也弱，这时候电影标题、类型、导演、国家、简介文本就特别关键。  
你项目里有文本语义召回，所以物品冷启动答起来会很自然。面试里建议明确区分“用户冷启动”和“物品冷启动”，这会显得很专业。

# 特征工程与经典推荐模型

## 你怎么做特征交叉？
短答：当前项目没有上显式深度交叉网络，更多是通过“用户历史加权 profile × 电影向量相似度”和 PPO 列表级特征间接做交互；如果升级到工业级，我会考虑 `Wide&Deep / DeepFM / DCN / DIN`。

展开：这题最忌讳硬吹。你现在项目里没有上 `DCN` 或 `FiBiNet` 这种显式交叉网络，所以不要说“我大量做了复杂特征交叉”。更好的答法是：  
当前项目用语义 profile 和候选向量相似度已经形成了一个轻量交互；  
PPO 中候选与已选列表的相似度，又构成了列表级交互；  
如果做工业升级，我会在精排里加入显式交叉网络，例如 `Wide&Deep` 记忆强规则、`DeepFM` 自动做二阶交叉、`DCN` 学有限阶显式交叉、`DIN` 做 target-aware 兴趣激活。

## `Wide&Deep`、`DeepFM`、`DCN` 分别解决什么问题？
短答：`Wide&Deep` 平衡记忆和泛化，`DeepFM` 降低人工交叉成本，`DCN` 更显式地学习有限阶交叉。

展开：  
`Wide&Deep` 适合讲“强规则 + 泛化”。  
`DeepFM` 强在“FM 负责低阶交叉，DNN 负责高阶交叉”，共享 embedding，减少人工 wide 特征。  
`DCN` 强在 cross layer 直接建模显式交叉，更适合面试里写公式。  
如果面试官问“你会优先哪个”，对实习岗很稳的回答是：排序 baseline 先 `DeepFM/DCN`，如果要行为序列感知再加 `DIN/DIEN`。

## DIN 为什么这么常考？
短答：因为它把“用户兴趣是多样的，而且和当前候选有关”这件事讲得最清楚。

展开：很多面经里都会问 DIN，因为它特别适合考察你是否理解推荐不是“一个用户一个固定向量”。DIN 的核心是 target-aware attention，也就是候选电影不同，激活的历史兴趣也不同。  
如果用户既喜欢悬疑也喜欢爱情片，面对《盗梦空间》和面对《爱乐之城》时，应该被激活的历史是不一样的。  
回答 DIN 时，把这层直觉说出来，比背结构更重要。

## 为什么召回和精排常常不用同一套模型？
短答：因为目标和计算预算不同，召回看覆盖和速度，精排看精度和交互建模能力。

展开：召回阶段面对的是全量库，必须非常快；精排阶段候选已经少很多，可以承受更复杂的模型和更多特征。  
工业里很少“一个模型吃全链路”，因为那样通常算不过来，也不一定最优。你项目里就是典型例子：召回用语义 embedding 和传统推荐，重排才上 PPO。

## 什么是样本粒度和特征粒度的错配？
短答：就是训练样本的定义和你喂进去的特征作用范围不一致，容易导致模型学偏。

展开：比如你训练的是用户-电影 pair 级样本，却塞进去很多列表级或未来信息特征，就可能造成泄露或目标错配。又比如你想做下一次点击预测，却把用户未来才发生的评分统计也放进来，那就是典型错误。  
这类问题在面试里很爱考，因为它能看出你是不是只会调包。

## 当前项目里最关键的特征是什么？
短答：如果分阶段看，召回最关键的是语义向量，重排最关键的是用户状态向量、候选电影特征和曝光/多样性信息。

展开：  
文本语义向量补足了“电影内容理解”；  
交互语义向量补足了“用户共现偏好”；  
PPO 阶段又额外引入 coverage 和 diversity，解决头部化和同质化。  
这个回答特别适合被问“你觉得自己项目里哪块最值钱”的时候用。

# Word2Vec、GloVe 与语义训练

## 你讲一下 Word2Vec 原理。
短答：Word2Vec 的核心是利用局部上下文共现，让语义相近的词学到相近向量；常见两种结构是 `CBOW` 和 `Skip-gram`。

展开：  
`CBOW` 是用上下文预测中心词；  
`Skip-gram` 是用中心词预测上下文。  
工业里更常提 `Skip-gram + Negative Sampling`，因为它在大词表上更高效。  
词向量不是监督标签学出来的，而是通过共现关系学出来的。  
放到你的项目里，文本侧 Word2Vec 本质是在电影标题、类型、导演、简介 token 序列上学 token embedding，再汇聚成电影向量；交互侧则是在用户正反馈电影序列上学 item embedding。

## 你项目里 Word2Vec 真训练了吗？
短答：训练了，但不是调用外部大包，而是本地做了一个轻量 `word2vec-like` 实现。

展开：这题可以直接结合代码答。你的项目里 `_train_word2vec_like` 会：  
构建 token vocab；  
随机初始化输入和输出 embedding；  
按窗口采样正上下文；  
再做负采样更新。  
维度默认是 `24`，窗口 `2`，轮数 `16`。  
所以它不是一句“用了 Word2Vec 思想”，而是真的训练了一个轻量版。

## 你讲一下 GloVe 原理。
短答：GloVe 用的是全局共现矩阵，不像 Word2Vec 更偏局部窗口采样；它本质是一个加权最小二乘的 log-bilinear 模型。

展开：Word2Vec 更偏“局部上下文预测”，GloVe 更偏“全局共现统计建模”。它先统计词和词之间的共现次数，再让词向量内积去逼近共现的对数关系。  
面试里一句很有用的话是：Word2Vec 更像预测式 embedding，GloVe 更像矩阵分解/统计式 embedding。  
你项目里的 `_train_glove_like` 也是本地轻量实现，会先构造共现矩阵，再优化 weighted least squares 目标。

## Word2Vec 和 GloVe 在这个项目里分别适合什么？
短答：Word2Vec 更适合局部共现和行为序列，GloVe 更适合把全局统计关系学稳；所以我两个都做了，对比实验更完整。

展开：这不是绝对划分，但面试时这么讲很顺。  
电影文本里，局部 token 共现和全局共现都有价值；  
用户行为序列里，局部上下文共现也非常关键。  
两个模型都跑一遍，既能增加可对比性，也让你面试时更有“算法选型”的话术，而不是单一路线。

## 为什么不用 TF-IDF 就够了？
短答：TF-IDF 更偏关键词匹配，语义泛化弱；Word2Vec/GloVe 能学到“词和词、电影和电影之间”的相似结构。

展开：如果只是做一个非常基础的内容推荐，TF-IDF 完全能跑。但你项目想体现一点算法深度，又希望在“悬疑≈惊悚”“诺兰≈烧脑科幻”这类关系上更灵活，词向量方案更有优势。再加上你项目里还手工做了同义词扩展，这个答法就更完整了。

## 词向量训练有没有负采样？
短答：Word2Vec-like 路线里有，GloVe 路线里没有用负采样，而是基于全局共现做加权回归。

展开：这题很细，正好体现你不是死背。你的本地 `word2vec-like` 实现会按频率分布采样若干负例，并更新输入/输出向量；而 `glove-like` 实现走的是共现矩阵和加权最小二乘，不需要负采样。  
很多面试官就是爱在这种地方卡你。

## 为什么 embedding 维度设成 24？
短答：这是当前项目数据量和表达能力之间的折中，小样本下维度太大反而容易过拟合和噪声放大。

展开：你的电影数据和行为数据规模都不大，所以 embedding 维度不宜盲目拉高。`24` 这种量级在单机、本地项目里很合理：  
足够表达基础语义；  
内存和计算都轻；  
不会像高维 embedding 一样在小样本下不稳定。  
面试里这种问题其实是在考你有没有超参直觉。

# PPO、Reward Model 与强化学习细节

## PPO 的效果到底怎么样？
短答：如果放在小候选集重排、且 reward 设计合理，PPO 通常能改善列表质量；但它不是银弹，效果高度依赖 reward、候选质量和样本量。

展开：这题千万别吹成“PPO 一上就大幅提升”。更稳的答法是：  
PPO 在当前项目里主要价值是把相关性之外的覆盖率、多样性、探索纳进重排决策；  
如果候选集本身很差，PPO 再强也救不了；  
如果 reward 设计不合理，策略可能学偏；  
如果样本太少，提升也会不稳定。  
这种“有边界感”的回答，在推荐和 RL 面试里会更加分。

## 为什么 PPO 适合这个项目？
短答：因为当前项目把 PPO 放在重排层，动作空间小、目标明确，比拿 PPO 直接做全量召回更合理。

展开：PPO 最怕动作空间太大、反馈太稀疏。你这里不是从全量电影库做策略选择，而是在召回后的候选集里做重排，所以问题规模被大幅缩小。加上你还有清晰的 `policy_score + coverage_bonus + diversity_bonus` 路线，这让 PPO 变成了一个“可解释、可控”的增强层，而不是难以落地的噱头。

## PPO 里的 `clip` 为什么重要？
短答：因为它限制新旧策略变化别太大，避免一次更新把策略带崩。

展开：当前项目里会算 `ratio = current_prob / old_prob`，然后看它是否超过 `1 ± epsilon` 的边界。超过后就不再让 actor 按这个样本继续大步更新。这个机制是 PPO 稳定性的核心。如果面试官问“为什么推荐里也要 clip”，你就答：推荐日志 noisy，策略太激进更容易震荡，所以 clip 更有意义。

## 你项目里的 advantage 怎么算？
短答：当前实现里是一个轻量近似，直接用 `reward - value` 作为 advantage。

展开：这题很容易被问得很细。你的项目没有完整实现 GAE，而是更轻量地把状态价值 `value` 作为 baseline，用 `reward - value` 近似 advantage。  
这在小项目里完全能讲通，因为重点是降低纯 reward 更新的方差。  
如果面试官问“工业里会更复杂吗”，你就答：会，常见会用多步回报和 GAE。

## 为什么 value 网络这么简单？
短答：因为当前项目目标是验证重排策略闭环，不是堆深网；value 先做轻量 baseline 足够支撑 advantage 估计。

展开：你项目里 value 就是 `state @ W_value + b_value` 这种轻量线性结构，这在大规模工业环境当然不够复杂，但在单机项目里有两个好处：  
训练稳定、容易解释；  
便于把重点放在 reward 设计、策略更新和重排逻辑本身。  
实习面试时，知道“为什么简单”通常比盲目追求复杂更重要。

## Reward Model 的损失怎么计算？
短答：如果做 LLM 或偏好学习场景，常见是 pairwise loss，比如 `-log(sigmoid(r_chosen - r_rejected))`。

展开：注意，这里要主动说明：当前项目没有单独训练 Reward Model。当前项目的 reward 来自规则映射和辅助奖励，不是学出来的 RM。  
但如果把项目扩展成 LLM 推荐排序或推荐解释对齐，就可以训练 RM。最常见的是成对偏好数据：给定两个候选，用户或标注者更喜欢 A，不喜欢 B，那么 reward model 输出两个分数 `r(A)` 和 `r(B)`，损失写成：
`L = -log sigma(r(A) - r(B))`  
这本质上是 Bradley-Terry 风格的偏好学习。  
如果面试官继续问 pointwise 也行吗，你可以说也可以，但 pairwise 更贴近“偏好比较”。

## Reward Model 的打分是怎么来的？
短答：来源通常有三种：人工偏好标注、业务行为日志映射、或高质量 teacher 模型打分。

展开：当前项目真实使用的是第二种，也就是把评分、收藏、点击映射成 reward。  
如果做更标准的 Reward Model，打分可以来自：
- 人工比较两种推荐结果哪个更好；
- 用业务行为构造偏好，比如完播/停留/高评分的推荐结果视为 chosen；
- 用更强模型做 teacher 评分，但这会把 teacher 偏差带进来。  
算法岗面试里，答出“打分不是凭空来的，而是来自偏好数据或行为代理信号”就很好。

## 为什么当前项目不单独训练 Reward Model？
短答：因为项目规模小、反馈形态简单，直接用业务规则映射 reward 更轻、更稳；单独 RM 会增加数据标注和训练复杂度。

展开：你当前项目主要是电影推荐闭环，不是 LLM 偏好对齐系统。用户行为数据本来就能直接转换成 reward，再加上 coverage/diversity 辅助奖励，已经足够支撑本地 PPO。  
如果单独引入 RM，就意味着要再定义偏好样本、训练标注体系、避免 reward hacking，这会显著增加工程复杂度。  
所以这个问题最好的答法是：不是不会，而是当前阶段没必要。

## 为什么强化学习里不用 `SFT -> RM -> PPO` 的完整 RLHF 三段式？
短答：因为当前项目不是大模型对齐任务，没有自然语言输出空间，也没有高质量偏好对比标注，直接规则奖励 + PPO 更匹配当前场景。

展开：这题其实是在考你会不会把 RLHF 和普通 RL 区分开。  
RLHF 那套 `SFT -> Reward Model -> PPO` 是为生成式模型对齐设计的；  
而你当前项目是结构化候选重排，动作空间和反馈定义都更清楚，直接用业务 reward 就足够。  
这句话一说，面试官基本会觉得你分得清任务边界。

## 为什么要引入 `ε-greedy`？
短答：为了探索，不然系统会越来越只推头部内容，数据越收越偏。

展开：这道题最好联系“反馈闭环偏差”。如果永远按当前最优策略排，长尾电影永远没机会曝光，也就永远收不到反馈，最后系统只会更头部化。`ε-greedy` 在当前项目里就是给低曝光内容一个受控探索通道。  
顺手补一句：探索比例不能太高，否则短期体验会掉。

# 评估指标与实验设计

## 你项目里用了哪些核心评估指标？
短答：`precision@k`、`recall@k`、`ndcg@k`、`coverage`、`diversity`。

展开：这题答完最好马上跟一句“每个指标看的是不同维度”。  
`precision@k` 看前 k 条里准不准；  
`recall@k` 看用户喜欢的目标有没有被找回来；  
`ndcg@k` 看命中位置；  
`coverage` 看系统是不是只集中推少数片；  
`diversity` 看单个列表是否太像。  
你的项目里这些指标是非常适合面试展开的。

## `Recall@K` 具体是什么意思？
短答：在当前 leave-one-out 设定里，就是目标电影是否出现在前 K 个推荐里；出现就是 1，不出现就是 0。

展开：因为你的离线验证是“每个用户留一个目标高分电影”，所以 `Recall@K` 在实现上其实退化成一个 hit 指标。  
严格地说，多标签场景下 recall 是“找回多少相关项 / 总相关项”，但你这里每个用户测试目标只有一个，所以 Recall@K 和 HitRate@K 非常接近。  
这类细节非常适合应对“你真的看过自己评估代码吗”的盘问。

## 为什么 `Precision@K` 会很小？
短答：因为当前每个用户只有一个测试目标，命中了也只是 `1/K`，所以数值天然不高。

展开：你项目里 `precision_at_k` 的实现就是：命中则 `1/k`，否则 `0`。这意味着即使模型命中了目标片，`Precision@10` 也只有 `0.1`。  
很多面试官会借这题看你是不是机械看数值高低。你要说明：Precision 的绝对值要结合任务定义解读，不能脱离样本设定谈高低。

## 为什么 NDCG 比 Recall 更能体现排序质量？
短答：因为 NDCG 会区分“排第 1 命中”和“排第 10 命中”，Recall 不区分。

展开：你的实现里如果目标电影排在第 `i` 位，`NDCG = 1 / log2(i + 1)`。这意味着越靠前命中，分越高。  
对推荐列表来说，这更符合用户真实浏览行为，因为靠前位置价值更大。  
面试时一句话总结：Recall 关心找没找回来，NDCG 关心找回来时排得靠不靠前。

## Diversity 是怎么计算的？
短答：当前项目里用的是列表内电影特征向量之间的平均余弦距离，越不相似，多样性越高。

展开：更具体地说，项目会遍历列表里两两电影，先算余弦相似度，再取 `1 - similarity` 作为距离，最后平均。  
这个设计很适合当前项目，因为你本来就有电影特征向量。  
如果面试官追问缺点，你就答：它只看向量空间距离，不一定完全等于用户主观感受的“多样性”。

## Coverage 是怎么计算的？
短答：看一段评估中一共推荐到了多少不同电影，再除以可推荐电影总数。

展开：Coverage 是系统级指标，不是单用户指标。它关心的是“平台生态有没有只压在少数头部片上”。  
面试里可以顺着说：Coverage 高不一定代表推荐就好，因为可能乱推长尾；但 Coverage 太低，通常说明系统过度头部化。

## 为什么离线评估用留一验证？
短答：因为推荐有时间性，用过去行为预测未来更合理，随机切分容易信息泄露。

展开：你项目里会把用户最后一个高分样本当测试目标，其余构成训练偏好。这是很标准、很适合小项目的 leave-one-out 方案。  
面试官如果问“为什么不用随机切分”，你就答：随机切分很可能把未来信息泄露到训练里。

## 你怎么看 AUC 和 GAUC？
短答：AUC 看 pairwise 排序能力，GAUC 是按用户或组聚合后的 AUC，更适合推荐和广告。

展开：虽然你项目当前主打的是 top-k 指标而不是 AUC，但面试里大概率还是会问。  
记住一句话：AUC 更适合点式二分类排序评估，GAUC 更适合用户分布不均衡的推荐场景。  
如果面试官问“你项目为什么没放 AUC”，就答：当前离线任务更接近 top-k 推荐和 leave-one-out 命中，因此 NDCG/Recall 更直接。

# 大模型、RAG、LoRA 与工程扩展

## 你这个项目有微调吗？
短答：当前没有大模型微调；真实训练的是本地词向量和本地 PPO。若扩到 LLM，我会优先考虑 `LoRA/QLoRA` 而不是全量微调。

展开：这题一定要直接把边界讲清。当前项目不属于“微调大模型”项目，所以不要假装做过 SFT 或 LoRA。  
但你可以补充：如果后续把 LLM 引入电影标签增强、候选重排或推荐解释，我会优先做参数高效微调，而不是全量微调。

## RAG 在这个项目里如果要落地，具体怎么做？
短答：先建电影知识库和评论知识库，再做 embedding 检索，最后把召回文档和用户问题一起喂给 LLM 生成推荐解释或问答结果。

展开：一个很自然的 RAG 版本是：
- 文档侧：电影简介、导演信息、豆瓣短评、标签、FAQ。
- 切块：按电影实体或评论段落切 chunk。
- 向量化：离线编码，建向量索引。
- 检索：按用户 query 或当前候选做相关文档召回。
- 生成：把召回文档和用户需求喂给 LLM，输出推荐理由或对话式建议。  
如果面试官问“RAG 和当前推荐主链路是什么关系”，你就答：更适合作为解释层和对话层增强，而不是直接替代召回主链路。

## RAG 里的 chunk 怎么切？
短答：优先按语义完整单元切，比如按电影实体、简介段、评论段，而不是生硬按固定字数切。

展开：这题很细，但现在越来越常见。  
如果 chunk 太大，检索噪声高；  
太小，又丢上下文。  
电影场景里非常适合按“电影基础信息块”“剧情简介块”“评论观点块”来切。  
再进一步可以做 overlap，避免关键信息刚好被切断。

## RAG 为什么需要 rerank？
短答：因为向量召回通常偏语义相似，不一定最贴合当前问题；rerank 可以在小候选上做更精细相关性判断。

展开：这其实和推荐系统的“召回-排序”一模一样。RAG 先用 embedding 找 top-k 文档，再用 cross-encoder 或 LLM 对候选文档重排，能显著提升相关性。  
如果面试官问“你项目里哪里能对应这个思路”，就答：现在的 `semantic_hybrid -> PPO rerank` 本质上也是这类分层思想。

## LoRA 为什么适合算法实习面试里的大模型问题？
短答：因为它是最常见、最实用的参数高效微调方案，几乎是“你了解大模型训练没有”的基础门槛。

展开：LoRA 的核心是冻结原模型参数，只学习低秩增量矩阵。这样训练参数更少、显存更省、吞吐更高。  
如果面试官问“为什么 LoRA 能 work”，就答：很多任务适配所需的更新本身近似低秩。  
这句话就很够用了。

## 为什么不直接全量微调？
短答：因为成本太高，而且很多推荐增强场景根本不需要改动全部参数。

展开：你的项目如果只是想让大模型学会“电影标签生成”“推荐理由生成”“小候选辅助重排”，没必要全量微调。全量微调更适合任务分布和底座差异很大、且资源充足的情况。  
对实习面试，这么答非常稳。

## `skill` 和 `MCP` 分别是什么？
短答：`skill` 更像可复用的任务模板/能力包，`MCP` 是让模型标准化接入外部工具和数据源的协议。

展开：如果面试官从大模型工程继续问到 `skill` 和 `MCP`，你可以这么答：  
`skill` 强调的是把一类稳定流程打包，比如“电影知识检索 + 理由生成 + 风险检查”；  
`MCP` 强调的是模型怎么标准化地调用数据库、向量库、检索服务、日志系统等外部资源。  
跟你项目结合时，可以说：如果后续做推荐运维助手，我会把“查实验指标”“查电影知识库”“调模型版本”这些能力通过 MCP 接出来，再把固定流程封装成 skill。

# PyTorch 高频细节拷打

## 你项目为什么不用 PyTorch？
短答：因为当前项目目标是本地单机可运行和闭环验证，`NumPy + SQLite + Flask` 足够；但 PyTorch 训练细节我仍然准备了，因为面试高频会追问。

展开：这题答法核心是“不是不会，而是当前阶段没必要”。如果你一上来就承认项目没用 PyTorch，但马上能把下面这些问题讲清，面试官通常不会纠缠。

## `model.train()` 和 `model.eval()` 的区别是什么？
短答：它们会影响 `Dropout`、`BatchNorm` 这类层的行为；训练时启用随机性和 batch 统计，推理时关闭或固定。

展开：这是 PyTorch 最高频细节题之一。  
`Dropout` 在 train 模式下随机失活，在 eval 模式下关闭。  
`BatchNorm` 在 train 模式用当前 batch 统计量，在 eval 模式用滑动均值方差。  
如果面试官问“为什么验证前一定要 `eval()`”，这就是答案。

## 为什么 `CrossEntropyLoss` 前面通常不要手动 softmax？
短答：因为 `CrossEntropyLoss` 内部已经做了 `log_softmax`，手动再做一次会导致数值不稳，还可能影响梯度。

展开：这是特别爱考的小坑。`nn.CrossEntropyLoss` 期望输入是 logits，不是概率。内部会做更稳定的 `log_softmax + NLLLoss`。  
如果你先 `softmax` 再送进去，数值稳定性和梯度都可能变差。

## `optimizer.zero_grad()` 一般放哪？
短答：通常放在每个 iteration 的前面，在 `loss.backward()` 之前。

展开：因为 PyTorch 的梯度默认是累加的，不清零就会把上一次 batch 的梯度叠到这一次上。  
标准顺序通常是：
`zero_grad -> forward -> loss -> backward -> step`。  
这类题非常细，但问的特别多。

## `torch.no_grad()` 和 `detach()` 区别是什么？
短答：`no_grad()` 是上下文级关闭梯度跟踪，`detach()` 是把某个 tensor 从当前计算图切断。

展开：  
`no_grad()` 常用于验证和推理，整体不建图；  
`detach()` 常用于某个中间结果不想继续反传，比如 target network、缓存 embedding。  
面试里能说出“一个是全局上下文，一个是局部张量”就很好。

## `view()` 和 `reshape()` 有什么区别？
短答：`view()` 要求内存连续，`reshape()` 更宽松，必要时会复制数据。

展开：如果 tensor 经过 `transpose` 或某些切片操作，可能不是 contiguous，这时 `view()` 会报错，而 `reshape()` 可能自动帮你处理。  
这类题就是典型 PyTorch 手感题。

## 为什么会有梯度爆炸？怎么处理？
短答：深层网络或长序列反传时梯度连乘变大导致更新失控，常用梯度裁剪、归一化、残差连接和更稳的初始化来缓解。

展开：这题既是深度学习基础，也是 PyTorch 面试常见代码细节题。  
如果面试官继续追问怎么写，PyTorch 里常见就是 `torch.nn.utils.clip_grad_norm_`。

## 激活函数常用哪些？分别什么场景？
短答：`ReLU` 是通用默认，`GELU` 常见于 Transformer，`Sigmoid` 多用于概率输出，`Tanh` 现在隐藏层较少单独使用。

展开：  
`ReLU` 简单高效，但有 dying ReLU 风险；  
`GELU` 更平滑，所以大模型和 Transformer 很常见；  
`Sigmoid` 适合二分类输出和门控；  
`Tanh` 零中心，但也会饱和。  
如果面试官问推荐排序 MLP 用哪个，答 `ReLU/GELU` 都可以，别答隐藏层全用 sigmoid。

## `DataLoader` 里的 `num_workers` 和 `pin_memory` 是做什么的？
短答：`num_workers` 控制数据加载并行进程数，`pin_memory` 让 CPU 到 GPU 的拷贝更快。

展开：这题属于“你是不是真的训练过模型”。  
`num_workers` 太小会喂不满 GPU，太大又可能导致进程开销或死锁。  
`pin_memory=True` 在 GPU 训练时通常能提高 host 到 device 的传输效率。  
虽然你项目当前没上 GPU，但这个问题算法实习会经常问。

## 如果让你用 PyTorch 重写当前 PPO，你会先改哪三件事？
短答：先把状态和动作特征封成 `Dataset`，再把 actor/value 改成标准 `nn.Module`，最后补齐 batch 化训练和日志监控。

展开：这题答法非常实战。你可以说：
- 用 `torch.Tensor` 批量化状态和候选特征；
- actor 和 critic 用 `MLP` 或更规范的线性层实现；
- 训练日志接 TensorBoard 或 W&B；
- 后续再考虑更标准的 advantage、GAE、minibatch PPO。  
这会让面试官觉得你不仅看得懂代码，也知道怎么演进。

# 面试最爱追问的“论文与模型视野”

## 常见推荐模型你会怎么按链路分类？
短答：召回看 `ItemCF/MF/双塔/图召回`，排序看 `Wide&Deep/DeepFM/DCN/DIN/DIEN/MMoE`，重排看规则重排、列表优化和 RL 重排。

展开：这类分类答法非常适合实习面试，因为条理最清楚。你不一定要把每个模型都讲深，但要能说出它属于哪一层、解决什么问题。

## 业界经典推荐论文你会重点准备哪些？
短答：至少准备 `Word2Vec`、`GloVe`、`Wide&Deep`、`YouTube DNN`、`DIN`、`SASRec`、`BERT4Rec`、`PPO`。

展开：  
`Word2Vec` 和 `GloVe` 解决 embedding 基础；  
`Wide&Deep`、`YouTube DNN` 是工业推荐经典；  
`DIN` 是国内面试高频；  
`SASRec`、`BERT4Rec` 是序列推荐经典；  
`PPO` 则是你项目里 RL 线最重要的一篇。  
对实习岗来说，准备到“知道问题、方法、优缺点和适用场景”就很够用了。

## 如果面试官问“你最熟的是哪三篇”怎么答？
短答：我会优先答 `Word2Vec`、`DIN`、`PPO`，因为它们分别对应语义建模、推荐排序和强化学习重排，和我的项目最贴合。

展开：这是很实战的答法，不求全，但求和自己项目强绑定。  
你项目里本来就有词向量训练和 PPO，答这三篇最自然。  
面试官顺着追问也更容易落到你熟悉的地方。
