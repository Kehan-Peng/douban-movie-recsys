# 项目深挖与系统设计

## 如何用一句话介绍你的项目？
15秒简答：这是一个基于 `Flask + SQLite` 的本地电影推荐系统，我把它从基础推荐升级成了“多路语义召回 + PPO 重排 + 离线评估 + 模型版本管理”的完整闭环。

3分钟详答：面试里通常是第一题，核心不是把技术名词堆满，而是快速让面试官知道项目边界、算法亮点和工程闭环。
我会先讲产品目标是电影推荐，再讲系统约束是本地单机运行，所以我没有直接上重型在线服务，而是用 `Flask + SQLite` 保证能跑通全链路。然后强调算法链路从基础协同过滤和内容推荐出发，升级到 `Word2Vec/GloVe` 语义召回，再用本地 `PPO` 做重排，并补上 `precision@k / recall@k / ndcg@k / coverage / diversity` 的离线评估。最后再讲工程侧做了经验池、批量训练、模型版本管理和后台实验页，这样面试官会觉得你不是只做了一个 demo，而是在做一个能迭代的推荐系统。

可能追问：
- 问：你最想突出哪三个关键词？答：本地可运行、推荐闭环、强化学习重排。

## 你的系统整体链路是怎样的？
15秒简答：链路可以概括成“数据采集与行为沉淀 -> 多路召回 -> 语义混合候选 -> PPO 重排 -> 离线评估与版本管理 -> 在线反馈再入池训练”。

3分钟详答：面经里经常会让你画系统图，这题本质是考你有没有系统化思维。我的项目前面是电影数据、评论数据和行为数据采集，行为里既有显式评分，也有点击、收藏这类弱反馈。召回层先做基础内容推荐、基础协同过滤，再做 `word2vec_content`、`glove_content`、`word2vec_cf`、`glove_cf` 这几路语义召回，组合成 `semantic_hybrid`。这一步主要解决候选集质量问题。然后重排层不是重新从全集选片，而是对候选集做 `PPO` 重排，并引入 `ε-greedy`、覆盖率奖励和多样性奖励。训练数据不会每来一条就更新，而是先进经验池，积累到一定量后批量训练，避免策略震荡。最后用后台实验页看离线指标，并把模型版本落盘和注册，支持回滚。

可能追问：
- 问：为什么要把 PPO 放在重排层而不是召回层？答：因为 RL 更适合在小候选集上做组合优化，直接做全量召回动作空间太大，训练也更不稳定。

## 为什么选 Flask + SQLite，而不是一开始就上大服务栈？
15秒简答：因为项目目标是单机可运行和快速验证算法闭环，`Flask + SQLite` 足够支撑小规模推荐实验，复杂度最低，可解释性也最好。

3分钟详答：这题其实在考选型意识。很多课程项目一上来就堆 Redis、Kafka、Spark、向量库，但如果数据量和并发量不匹配，反而显得只会堆技术栈。我的项目强调本地单机运行，所以选型优先级是“能稳定跑通推荐和实验闭环”，不是“模拟大厂全部基础设施”。`Flask` 足够承载页面、接口和后台管理；`SQLite` 足够承载电影数据、行为数据、经验池和模型元数据。这样做的好处是开发快、部署简单、便于面试时讲清楚。缺点我也会主动承认，比如并发能力有限、事务吞吐有限、不适合真实超大规模在线服务。这样回答会让面试官觉得你是基于场景做权衡，而不是不知道工业系统长什么样。

可能追问：
- 问：如果面试官说这个架构太轻怎么办？答：我会说明这是单机实验闭环，后续迁移到工业级可以把存储、特征服务和在线推断拆分出去。

## 为什么你的项目采用“多路召回 + 重排”的结构？
15秒简答：因为推荐系统的核心矛盾是“全量物品太多，单模型很难兼顾召回广度和排序精度”，所以先用多路召回保覆盖，再用重排保精度和多样性。

3分钟详答：这是推荐系统面试里的绝对高频题。召回层负责从大量电影里先筛出几百或几十个候选，追求的是覆盖率、召回率和候选质量；排序或重排层负责在小候选集上做更精细的打分和组合优化，追求点击、满意度、多样性、长期价值等。我的项目里基础内容推荐和协同过滤提供传统 baseline，`Word2Vec/GloVe` 召回补充语义和序列共现信息，最后 `PPO` 在候选集上做重排。这样一是更像工业架构，二是方便做 ablation，对比每一层带来的增益。面试里如果你直接说“我用了 PPO 推荐电影”，面试官很容易追问动作空间和训练稳定性；如果你说明是做重排，他们通常会觉得设计更合理。

可能追问：
- 问：多路召回之间是相加还是去重拼接？答：先做去重拼接，再按业务规则或打分融合进入后续重排。

## 为什么 PPO 只做重排，不直接承担全链路推荐？
15秒简答：因为全链路推荐动作空间太大、反馈太稀疏、训练成本高，而重排只面对有限候选集，更适合 PPO 做策略优化。

3分钟详答：RL 在推荐里最容易被问的就是“你为什么不用它直接做全链路”。核心答案是工程可行性和样本效率。对全量电影库来说，动作空间可能上千上万，用户真实反馈又很稀疏，on-policy 的 `PPO` 很难稳定学到有效策略。而重排阶段候选集已经被召回层缩小，比如项目里取 `top_n * 3` 的候选池，这时策略网络只需要在更小、更高质量的集合里决定展示顺序，问题难度下降很多。同时重排更容易引入覆盖率、多样性、长期收益这些“列表级目标”。所以我把 RL 放在它最有优势的位置，而不是为了炫技让它承担不擅长的工作。

可能追问：
- 问：那为什么不用监督学习排个序就行？答：监督排序更擅长点式相关性预测，而 RL 更适合把列表级奖励和长期反馈纳入目标。

## 你的候选集是怎么构建的？
15秒简答：候选集主要来自 `semantic_hybrid`，它融合了文本语义和交互语义两类召回；如果不够，再回退到 `baseline_hybrid` 和热门片补齐。

3分钟详答：候选集构建是很容易被深挖的项目细节。我这里不是直接用一个单点算法召回，而是先做多路候选。文本侧用电影标题、类型、导演、国家、简介生成 `word2vec` 和 `glove` 风格的语义向量，交互侧把用户正反馈序列当作 item sequence，训练 `item2vec` / 共现式嵌入。然后把这几路分数融合成 `semantic_hybrid`。如果用户无历史、召回不足或语义路结果太少，就降级到基础内容推荐、协同过滤和热门片补齐。这样设计可以避免单路召回偏科，也方便面试时解释“为什么召回要多样化”。

可能追问：
- 问：为什么候选池设置成 `top_n * 3`？答：这是精度和计算量之间的折中，太小会限制重排空间，太大会增加计算和噪声。

## `semantic_hybrid` 的核心思想是什么？
15秒简答：核心是把“电影文本相似”与“用户交互共现”两种语义信号合在一起，避免只看文本或只看行为导致的偏差。

3分钟详答：很多推荐项目只做内容相似或者只做协同过滤，我的 `semantic_hybrid` 想解决的是二者各自的短板。文本语义能理解电影题材、导演、风格和简介关键词，适合冷启动和内容补全；交互语义能从用户共同观看、评分、收藏序列里学到“经常一起被喜欢”的共现关系，更接近真实偏好。两者结合后，既能解释用户为什么会喜欢某一类电影，也能利用行为数据捕捉隐式关联。面试里可以把它讲成“内容理解 + 行为共现”的混合召回。对于你这个豆瓣项目来说，这比单纯讲协同过滤更有层次，也自然能衔接到后面的 RL 重排。

可能追问：
- 问：两路分数怎么融合？答：可以做加权融合，权重由离线实验调，项目里也是按多路得分组合后再排序。

## 为什么在这个项目里选 Word2Vec/GloVe，而不是直接上 BERT？
15秒简答：因为当前项目强调本地单机运行和小规模数据，`Word2Vec/GloVe` 训练轻、部署轻、解释性更强，性价比比 BERT 更高。

3分钟详答：这是非常典型的“算法选型”问题。`BERT` 句向量的表达力确实更强，但代价是模型更重、推理更慢、工程复杂度更高，而且你的电影简介和行为数据规模还没有大到一定要靠大模型向量才能起效。相对来说，`Word2Vec/GloVe` 在小语料、小样本下更容易训练和解释，也能同时覆盖文本侧和交互侧。更关键的是，面试里你不能只说“BERT 更强”，你要说明“为什么当前阶段不划算”。我的答案会是：先用轻量语义向量把语义短板补齐，等数据量和算力上来后，再把 BERT 作为离线预编码特征接到召回或排序里，而不是一开始就替换全链路。

可能追问：
- 问：如果一定要引入 BERT，你会怎么接？答：我会离线预计算电影文本向量，把它作为召回或排序特征，而不是在线实时跑大模型。

## 用户状态向量在你的 PPO 重排里怎么定义？
15秒简答：状态向量主要编码用户历史偏好、近期行为和部分语义特征，本质上是在给策略网络提供“当前用户兴趣画像”。

3分钟详答：RL 推荐里状态定义比模型名更重要。我的项目不是直接把原始日志喂给 PPO，而是先把用户历史转换成状态向量。里面会包含用户喜欢过的电影、评分倾向、行为强度，以及由电影特征和语义标签聚合出来的偏好表征。这样做有两个目的：第一，降低原始行为序列的噪声；第二，把策略学习问题转成“给定用户状态，如何在候选集上做更优排序”。如果面试官继续问，你可以补一句，工业场景里状态通常还会加入时间上下文、设备、页面位置、会话信息和实时特征，而这个项目保留的是核心思路。

可能追问：
- 问：为什么不用完整序列直接做策略输入？答：因为项目规模较小，先做结构化状态向量更稳定，也更容易训练和解释。

## 在你的项目里，动作空间怎么定义？
15秒简答：动作不是从全量电影中盲选，而是在候选池内对电影进行排序决策，本质上是“受限动作空间下的重排”。

3分钟详答：很多同学说自己用了 RL，但一被问动作空间就答不清楚。我的项目里动作空间不是整个电影全集，因为那样维度太大且样本极稀疏。更合理的定义是：先由召回层提供候选电影，再由 PPO 根据用户状态和候选特征给每个动作分配概率，进而决定重排顺序。代码里会先构建电影特征矩阵，再用策略网络输出 `policy_score`，之后叠加 coverage bonus 和 diversity bonus 形成最终的 `rl_score`。这个定义很符合工业里的 slate reranking 思路，也更容易解释 PPO 为什么在这里能落地。

可能追问：
- 问：动作是选一个 item 还是选一个列表？答：项目实现上更接近逐步选 item 的列表构造，但目标是优化最终列表效果。

## 你的奖励函数是怎么设计的？
15秒简答：主奖励来自评分、收藏、点击等用户反馈，辅助奖励包括覆盖率和多样性，用来避免只推热门且高度相似的片子。

3分钟详答：奖励设计是 RL 推荐最常见的深挖点。我的主奖励来自行为类型和评分强弱，比如高评分是更强的正反馈，收藏和点击给较弱正反馈。然后我额外设计了两个辅助项。一个是覆盖率奖励，曝光少的电影会拿到更高 bonus，避免系统长期只推热门头部；另一个是多样性奖励，如果候选电影和已经选出的片子太像，它的 bonus 会下降。这样做的逻辑是：推荐系统不只要追求短期点击，还要兼顾列表体验和内容生态。面试里你也要主动说明 reward shaping 的风险，比如辅助奖励权重过大时会伤害相关性，所以最终权重要靠离线实验校准。

可能追问：
- 问：为什么点击奖励比评分小？答：因为点击是弱反馈，噪声更大，评分更能代表真实满意度。

## 为什么要在重排里加入 `ε-greedy`？
15秒简答：因为纯 exploitation 会让系统越来越保守，`ε-greedy` 能给长尾和新物品少量探索机会，帮助收集更多真实反馈。

3分钟详答：强化学习和推荐系统共有的核心问题之一就是探索与利用。只按当前最高分推荐，看起来短期指标可能不错，但系统会越来越偏头部和既有兴趣，导致新片和长尾片没有曝光，也拿不到训练数据。我的项目里在重排阶段加入 `ε-greedy`，当触发探索时，会按覆盖率相关权重随机挑选候选，而不是永远选最高 `rl_score` 的电影。这样做的好处是策略不会过早收敛到局部最优，也能让经验池里持续出现新的样本。对于面试来说，这能很好地说明你知道 RL 不是只会背 PPO 名字，而是真的考虑了线上反馈闭环。

可能追问：
- 问：探索比例太大有什么问题？答：会伤害短期体验和指标，所以一般只保留小比例探索。

## 覆盖率奖励和多样性奖励为什么都要加？
15秒简答：覆盖率解决“总推热门”，多样性解决“推得都很像”，两者优化的是两个不同维度的列表质量。

3分钟详答：很多人会把覆盖率和多样性混在一起，但它们不是一回事。覆盖率偏全局，关注系统整体是否只把流量集中到少量头部电影上；多样性偏列表级，关注一屏推荐里的内容是否过于相似。比如全是诺兰电影可能相关，但用户体验未必最好。我的项目里 coverage bonus 通过曝光次数反向加权，多样性 bonus 通过候选特征和已选电影特征的相似度计算。两个奖励一起加，能让推荐结果既不过度头部化，也不会内容同质化。面试里说清这个区别，通常会让面试官觉得你对推荐指标理解更立体。

可能追问：
- 问：这两个奖励会不会损伤准确率？答：会有 trade-off，所以权重不能拍脑袋定，要靠离线实验找平衡点。

## 为什么经验池要批量训练，而不是每条反馈都立刻更新？
15秒简答：因为逐条更新噪声大、策略容易震荡，批量训练更稳定，也更像真实系统里的 mini-batch 更新方式。

3分钟详答：这题考的是训练稳定性意识。用户一条行为不一定可靠，比如误点、随手收藏、评分延迟都可能带来噪声。如果每来一条反馈就更新一次 PPO，策略很容易被偶然样本带偏，导致线上行为波动很大。我的做法是先把体验写入 `rl_experience`，当样本数达到阈值后再做一次批量训练。这样可以让梯度更稳定，也方便记录本次训练对应的模型版本和平均奖励。你还可以补一句，工业里常见做法也是流式写日志、批处理训练或近线更新，而不是每个事件都立刻改模型。

可能追问：
- 问：那实时性会不会变差？答：会有一点，但在当前项目里我优先保证稳定性，后续可以做近实时小批更新。

## 你的模型版本管理是怎么设计的？
15秒简答：训练后模型权重会本地落盘，同时在 SQLite 注册元数据，支持查看、回滚和清理旧版本。

3分钟详答：很多项目只训练模型，不管理模型版本，这在面试里很容易被认为缺少工程闭环。我的项目训练完成后，会把 PPO 权重保存到本地目录，再把版本号、训练时间、关键指标等写入数据库。这样后台能展示版本列表，支持当前激活版本查看、历史版本回滚和旧版本清理。这个设计的价值有三点。第一，实验可追溯，不会出现“效果变好但不知道用了哪个模型”。第二，回滚方便，线上指标波动时能快速退回稳定版本。第三，和经验池处理状态能关联起来，知道哪些样本是由哪个版本消费的。面试时讲这一段，很容易把你和只会调包跑模型的人区分开。

可能追问：
- 问：版本号怎么命名更好？答：可以用时间戳加算法标签，既唯一又方便追踪。

## 你是怎么做离线评估的？
15秒简答：我用留一验证构造样本，评估 `precision@k`、`recall@k`、`ndcg@k`、`coverage` 和 `diversity`，同时对比 baseline、语义召回和 PPO 重排。

3分钟详答：离线评估是项目可信度的关键。我的做法是先从用户历史评分里取正反馈样本，比如高分电影；然后对每个用户做 leave-one-out，把最后一个正样本作为测试目标，剩下行为构成训练偏好。接着用不同算法给出 top-k 推荐，看目标电影是否出现在结果里，并计算 `precision@k`、`recall@k`、`ndcg@k`。此外，为了体现推荐系统不只是“命中率机器”，我还评估了覆盖率和多样性。这样就能比较基础热门、基础内容、基础协同过滤、语义混合以及 PPO 重排之间的差异。面试里最好主动承认局限性，比如离线指标不能完全代表真实线上收益，但它至少能支持算法间相对比较。

可能追问：
- 问：为什么用 NDCG 而不是只看 Recall@K？答：因为 NDCG 会考虑命中位置，越靠前命中价值越大，更接近真实排序质量。

## 为什么选择留一验证，而不是随机切分？
15秒简答：因为推荐数据有时间性，留一验证更符合“用过去预测未来”的实际场景，随机切分容易信息泄露。

3分钟详答：这题本质上考你有没有数据科学意识。对推荐系统来说，用户行为是有时间顺序的。如果随机把用户看过的电影打散进训练集和测试集，很可能出现“模型用未来行为预测过去行为”的信息泄露。留一验证虽然简单，但更贴近真实线上情境：拿用户过去的偏好去预测下一次可能喜欢什么。我的项目里就是保留用户最后一个高分样本当目标，其余行为构成用户画像。这样做在小项目里实现简单、可解释性强，很适合面试场景。你也可以补充说，工业上还会做更严格的按时间窗口切分。

可能追问：
- 问：留一验证的局限是什么？答：测试样本较少，且过于依赖最后一个行为，适合作为小项目离线基线但不等于完整评估。

## 你的项目怎么处理冷启动？
15秒简答：用户冷启动走热门片和内容语义，物品冷启动优先依赖电影文本特征，避免完全依赖协同过滤。

3分钟详答：冷启动是推荐面经里高频到几乎必问的问题。我的项目把用户冷启动和物品冷启动分开看。用户没有历史行为时，协同过滤和交互嵌入基本失效，这时就回退到热门电影和内容语义召回，比如根据电影类型、导演、简介文本做相似推荐。对于新电影，由于还没有足够曝光和评分，行为特征也不足，所以主要依赖文本和结构化元信息生成特征，先让它有机会进入候选池。等收集到点击、收藏、评分之后，再逐渐让行为特征和 PPO 反馈接管更细的排序。面试里如果你能把用户冷启动和物品冷启动分开说，答案会显得更完整。

可能追问：
- 问：新用户第一屏推荐最常见的做法是什么？答：热门榜、品类精选、基于少量注册信息的人群先验是最常见方案。

## 你如何向面试官证明这个项目“真的有效”？
15秒简答：我会用“三层证据”来证明：离线指标对比、实验页可视化趋势、以及算法和工程逻辑上的可解释增益。

3分钟详答：面试中最怕的是“我感觉效果更好了”。所以我会把证据拆成三层。第一层是定量证据，用离线指标对比 baseline、语义混合和 PPO 重排。第二层是实验过程证据，比如后台实验页能展示多次快照、不同算法曲线和模型版本。第三层是机制解释，比如为什么语义召回能改善冷启动，为什么 PPO 重排能改善多样性和覆盖率。这样即使绝对数据规模不大，也能说明优化方向是自洽的。对面试官来说，小项目不一定要求你做到真实亿级流量，但很看重你是否知道该怎么科学地证明一项优化有效。

可能追问：
- 问：如果离线指标提升不明显怎么办？答：我会诚实说明，并分析是数据量、特征表达还是奖励设计的问题，而不是硬说效果很好。

## 这个项目最容易被质疑的点是什么？你怎么回应？
15秒简答：最容易被质疑的是数据规模小、RL 是否必要、单机架构是否太轻；我的回应是把它定位成“可运行的推荐闭环原型”，重点展示方法论和可演进性。

3分钟详答：好的项目介绍不是只讲优点，也要知道风险点。我的项目最容易被问三件事。第一，数据规模小，为什么还做 PPO？我的回答是 PPO 只做重排，不做全集推荐，而且目的是验证策略优化思路。第二，为什么不用工业大模型和分布式架构？我的回答是当前约束是单机运行和快速迭代，所以优先把闭环打通。第三，指标是否有统计显著性？我的回答是小项目更多看相对趋势和方法合理性，不能夸大结论。面试里能主动指出局限，反而会显得你更成熟。

可能追问：
- 问：如果面试官说 RL 有点 overkill 怎么办？答：我会承认这是带有探索性的增强层，但它确实适合讲列表级目标和长期反馈优化。

## 如果把项目升级到工业级，你会怎么演进？
15秒简答：我会把系统拆成特征、召回、排序、实验平台四层，引入更稳定的双塔或序列模型做召回，用监督排序做主排序，RL 保留在重排或策略层。

3分钟详答：这题本质是考你有没有工业视角。我的升级路线会分三步。第一步，数据和服务解耦，把离线样本生成、特征服务、在线推荐接口拆开，引入缓存、消息队列和更合适的存储。第二步，算法分层更明确，召回侧上双塔或图召回，排序侧上 `Wide&Deep / DeepFM / DIN / DCN / MMoE` 这类工业常见模型，RL 保留在重排或流量分配层。第三步，引入更完整的实验系统，比如 AB 测试、特征版本管理、实时监控和告警。对于大模型，可以先做文本增强、query rewrite、理由生成和少量高价值场景重排，而不是直接替代整条主链路。

可能追问：
- 问：工业里 RL 最适合放在哪？答：通常更适合放在重排、流量调控、长期价值优化这些策略层，而不是直接替代所有召回排序。

## 你为什么说这是“推荐闭环”而不是“推荐 demo”？
15秒简答：因为它不仅有推荐结果，还包含数据采集、行为回流、模型训练、版本管理、实验对比和评估，形成了从反馈到优化的闭环。

3分钟详答：很多项目只做到“给出一个推荐列表”，严格说那只是一个推荐 demo。我的项目之所以能叫闭环，是因为用户行为会继续写回数据库，经验会入池，达到阈值后触发训练，训练产出新模型，模型会注册版本并用于新一轮推荐，后台还能看到实验对比结果。闭环思维是推荐系统岗位非常看重的能力，因为线上系统不是一次性跑完就结束，而是持续接受反馈、持续迭代。你在面试里把这点讲出来，面试官通常会更愿意深挖，因为这说明你理解推荐系统不是单纯“训练一个模型”。

可能追问：
- 问：闭环里最关键的一环是哪一环？答：我认为是高质量反馈回流，因为没有可靠反馈，后面的训练和优化都很难成立。

## 为什么这个项目适合拿来面机器学习/推荐/强化学习实习岗？
15秒简答：因为它同时覆盖了经典推荐、语义建模、强化学习重排、离线评估和工程闭环，能应对大部分实习岗高频追问。

3分钟详答：实习岗位面试很少要求你真的做过亿级系统，但会看你是否有完整的问题意识。这个项目的好处是：机器学习岗可以聊特征、指标、优化和泛化；推荐岗可以聊召回、排序、冷启动、评估和工程分层；强化学习岗可以聊状态、动作、奖励、探索和 PPO；大模型相关岗位还可以自然过渡到为什么当前没直接上 LLM、未来怎么演进到 LLM4Rec。所以这不是一个只能应付单一方向的项目，而是一个很适合做“一个项目打多种岗位”的表达模板。

可能追问：
- 问：如果只给一分钟介绍，你会怎么裁剪？答：我会保留项目定位、算法亮点和闭环能力三部分，其余细节留给追问。

# 机器学习与深度学习基础

## 什么是过拟合？在推荐项目里怎么处理？
15秒简答：过拟合就是模型把训练数据噪声也学进去了，训练好、泛化差；推荐里常见处理方式是正则化、早停、降维、特征约束和更合理的验证切分。

3分钟详答：这是机器学习最基础也最常被追问的概念。过拟合通常表现为训练集指标很好，但验证集和测试集明显变差。推荐场景里更容易过拟合，因为行为稀疏、特征维度高、长尾严重，而且一些特征高度 id 化。处理方法通常有几类。第一类是模型层面，比如 `L1/L2` 正则、dropout、减少参数量。第二类是数据层面，比如更合理的负采样、按时间切分验证集、去掉泄露特征。第三类是特征层面，比如限制稀疏 id 直接记忆、加入统计特征和共享 embedding。放到你的项目上，就是不能只盯着离线命中，而要看不同算法在留一评估上的稳定性。

可能追问：
- 问：欠拟合和过拟合怎么快速区分？答：训练集都学不好更像欠拟合，训练集很好但验证集差更像过拟合。

## L1 和 L2 正则有什么区别？
15秒简答：`L1` 倾向产生稀疏解，适合做特征选择；`L2` 倾向让权重整体变小但不一定为零，更常用于稳定训练。

3分钟详答：面试里这题考你是否理解正则化背后的几何意义。`L1` 在目标函数里加权重绝对值和，优化后很多参数会被压成 0，所以有天然的特征选择效果；`L2` 加的是平方和，会更平滑地惩罚大权重，让模型不至于过度依赖少数特征。实际工程里，深度学习模型更常用 `L2` 或 weight decay，因为优化更稳定；线性模型和高维稀疏特征场景里，`L1` 更有吸引力。推荐系统中，如果你有很多高维稀疏交叉特征，`L1` 可以帮助裁剪无效特征；但如果是 embedding 和深度网络，通常 `L2` 更常见。

可能追问：
- 问：为什么 L2 更常见？答：因为它可导、优化稳定，而且不会像 L1 那样在 0 点附近带来更强的不连续性。

## BatchNorm 和 LayerNorm 有什么区别？
15秒简答：`BatchNorm` 沿 batch 维做归一化，适合 CNN；`LayerNorm` 沿特征维做归一化，更适合 Transformer 和序列建模。

3分钟详答：这题是深度学习岗位高频基础题。`BatchNorm` 依赖同一个 mini-batch 里的样本统计量，所以在 batch 较大、分布相对稳定的视觉任务里很有效；但在序列模型、在线推理和 batch 很小时，它容易不稳定。`LayerNorm` 是对单个样本内部的隐藏维做归一化，不依赖 batch 大小，所以特别适合 Transformer。你在回答时可以直接联系大模型：为什么 LLM 几乎都用 `LayerNorm/RMSNorm` 而不是 `BatchNorm`？因为自回归推理和序列建模不适合依赖 batch 统计量。这样就从一个基础题自然过渡到了 LLM。

可能追问：
- 问：为什么推荐里的序列模型更偏向 LayerNorm？答：因为序列长度和 batch 变化大，LayerNorm 对这类场景更稳。

## Dropout 的作用是什么？为什么推理时不用？
15秒简答：Dropout 通过随机屏蔽部分神经元来降低共适应，提升泛化；推理时要关掉，是为了使用完整网络并保持输出稳定。

3分钟详答：Dropout 的核心思路是训练时随机把一些神经元置零，让模型不能过度依赖某些局部通路，等价于对很多子网络做近似集成。这对小数据和容易过拟合的场景尤其有效。推理阶段如果还随机屏蔽神经元，输出会不稳定，所以通常会关闭 dropout，并用训练时的期望缩放保证分布一致。推荐场景里，dropout 常用于 MLP 排序模型或 embedding 后的深层网络，但对非常稀疏、非常浅的模型未必总有收益。回答时别只背定义，最好补一句“它的本质是正则化，而不是提升表达能力”。

可能追问：
- 问：dropout 一定有用吗？答：不一定，数据够大、模型够稳定时收益可能有限，甚至会拖慢收敛。

## 为什么分类里经常用交叉熵，而不是 MSE？
15秒简答：因为交叉熵和概率分布匹配更自然，梯度更有效，特别适合 softmax 或 sigmoid 输出；MSE 在分类里容易带来梯度问题。

3分钟详答：从本质上看，交叉熵衡量的是预测分布和真实分布之间的差异，因此非常适合分类问题。对于二分类 CTR、点击预测、喜欢/不喜欢建模，模型输出的就是一个概率，交叉熵天然对应最大似然估计。相比之下，`MSE` 更适合回归，因为它把标签当连续值处理。用在分类上虽然不是绝对不行，但在 sigmoid 饱和区间里梯度会更弱，优化通常不如交叉熵直接。推荐系统里点击率建模、曝光点击预测、转化预测一般默认就是 logloss，也就是交叉熵损失。

可能追问：
- 问：评分预测为什么常用 MSE？答：因为评分是连续值或近似连续值，本质更接近回归问题。

## AUC、GAUC 和 LogLoss 分别在看什么？
15秒简答：AUC 看排序能力，GAUC 是按用户或组加权后的 AUC，更适合推荐；LogLoss 看概率校准和整体拟合程度。

3分钟详答：这是推荐岗常见指标题。AUC 反映的是随机抽一个正样本和一个负样本，模型把正样本排在前面的概率，所以它更关注排序而不是绝对概率。GAUC 则是在用户、query 或流量组内先算 AUC，再按曝光量加权平均，能减少“一个大用户群把整体 AUC 带偏”的问题，所以在推荐和广告更常见。LogLoss 又不同，它对预测概率是否靠谱非常敏感，模型把一个本来很确定的正样本预测得很低，会受到较大惩罚。所以实战里通常不会只看一个指标，而是会把 AUC/GAUC 和 LogLoss 一起看。面试里如果能说出“高 AUC 不一定代表概率校准好”，就已经比只背概念更进一步了。

可能追问：
- 问：为什么推荐里常看 GAUC？答：因为用户分布不均衡，直接算整体 AUC 可能被头部用户或头部流量主导。

## Precision、Recall、F1、NDCG 在推荐里怎么理解？
15秒简答：Precision 看推荐出去的有多少是对的，Recall 看该命中的命中了多少，F1 是二者折中，NDCG 额外考虑命中位置。

3分钟详答：这些指标虽然都和“准不准”有关，但关注点不同。`Precision@K` 是推荐列表里有多少是真正相关的，强调前几条够不够准；`Recall@K` 是用户实际喜欢的内容有多少被你找回来了，强调不要漏太多；`F1` 是 precision 和 recall 的调和平均，更适合综合平衡；`NDCG@K` 则考虑命中的位置，越靠前命中价值越大，所以更接近用户真实浏览行为。对推荐系统尤其是 top-k 场景来说，NDCG 往往比纯 recall 更有解释力。你的项目里就同时评估 recall 和 NDCG，这比只报一个指标更完整。

可能追问：
- 问：为什么排序任务通常更偏爱 NDCG？答：因为排序结果的前部位置价值更高，NDCG 能体现这一点。

## 样本不平衡怎么处理？
15秒简答：可以从重采样、loss 加权、阈值调整、难例挖掘和更合理的评估指标几个方向处理。

3分钟详答：推荐和广告数据天然是极不平衡的，因为曝光很多、点击很少、转化更少。直接训练时，模型很容易学成“全预测不点”也能获得不错表面准确率。所以首先要换指标，别只看 accuracy；其次要做样本层处理，比如负样本下采样、正样本过采样、focal loss 或 class weight；再次要注意线上推断时分布偏移，训练时下采样了负样本，推理阈值可能不能照搬。对于推荐项目来说，曝光未点击通常很多，关键不是机械地把负样本采满，而是让训练集和真实业务目标一致。

可能追问：
- 问：accuracy 为什么不适合极不平衡数据？答：因为全预测负类也可能很高，掩盖模型对正类的真实区分能力。

## 推荐里的负样本一般怎么采？
15秒简答：常见有随机负采样、按曝光日志采样、按时间窗口采样和难负样本采样；是否把未曝光样本当负例要非常谨慎。

3分钟详答：负采样是推荐面经超级高频题。最容易犯的错是把用户没点过的所有东西都当负样本，但很多内容只是没被曝光，不代表用户不喜欢。更合理的做法是优先从曝光未点击里采负样本，因为这些才是真正用户“看到了但没选”。如果没有曝光日志，才会退化成从未交互物品里随机采一些伪负样本。工业里还会做难负样本挖掘，比如从召回结果里找和正样本很像但最终没点的样本，让模型学会更细的区分。你的项目数据规模较小，回答时可以说明：负采样的核心不是数量，而是尽量贴近真实决策边界。

可能追问：
- 问：为什么难负样本更有价值？答：因为它更接近模型容易混淆的边界，能提升排序区分能力。

## Embedding 的作用是什么？
15秒简答：Embedding 的本质是把高维稀疏离散特征映射成低维稠密向量，既降维，又能学习语义相似性。

3分钟详答：推荐系统里大量特征都是离散 id，比如用户 id、电影 id、导演、类型、关键词。如果直接 one-hot，维度会极高且难以表达相似关系。Embedding 把每个离散 token 映射成一个可训练向量，使模型可以在连续空间里学习“谁和谁更像”。比如两个经常被同类用户喜欢的电影，它们的 embedding 可能更近。你的项目里 `Word2Vec/GloVe` 本质也是一种 embedding 学习。面试里如果只说“embedding 是降维”，答案太浅；要补上“它还能承载统计共现和语义关系”，这样就更像推荐岗位的回答。

可能追问：
- 问：embedding 维度越大越好吗？答：不一定，维度太大容易过拟合，也会增加存储和计算成本。

## 连续特征和离散特征通常怎么处理？
15秒简答：离散特征多用 embedding，连续特征多做归一化、分桶或直接输入 MLP，关键看模型类型和特征分布。

3分钟详答：推荐特征工程里最典型的就是“id 类特征”和“统计数值特征”同时存在。离散特征像用户 id、电影 id、导演、类型，常见做法是 one-hot 后 embedding 或直接哈希到 embedding 表。连续特征像评分均值、热度、时长、曝光次数，通常需要标准化、归一化或分桶。比如树模型更喜欢离散分桶后的特征，深度模型则可以直接接连续值。回答时最好补一句，连续特征是否要 log 化、分桶化，取决于分布是否长尾，这也是推荐里常见的工程经验。

可能追问：
- 问：为什么很多连续特征会做 log 变换？答：因为曝光、点击、频次这类特征常呈长尾分布，log 可以压缩尺度。

## Adam 和 SGD 的区别是什么？
15秒简答：`SGD` 简单稳定、泛化往往更好；`Adam` 自适应学习率、收敛更快，对稀疏和初期训练更友好。

3分钟详答：`SGD` 只有统一学习率，更新方向更“朴素”，有时收敛慢，但泛化性能常不错。`Adam` 结合了一阶矩和二阶矩估计，相当于对不同参数使用不同步长，所以在稀疏特征、复杂深网和前期训练阶段通常更省心。推荐系统里由于 embedding 大量存在、特征稀疏明显，`Adam` 往往更常见。但这不代表它永远更好，一些大规模训练后期会切回 `SGD` 或调小学习率以求更稳的泛化。面试里最好的回答不是“Adam 更高级”，而是“它们是收敛速度和泛化特性的权衡”。

可能追问：
- 问：为什么 Adam 可能泛化不如 SGD？答：因为它自适应过强时可能更快拟合训练分布，落到更尖锐的极小值附近。

## 学习率为什么这么重要？常见策略有哪些？
15秒简答：学习率决定每次参数更新步子有多大，太大不收敛，太小收敛慢；常见策略有 warmup、step decay、cosine decay 和自适应优化器。

3分钟详答：很多训练问题表面上看像模型不行，其实是学习率没调好。学习率太大时损失震荡甚至发散，太小时训练非常慢，还可能卡在差的局部最优。工业训练里常见做法是先 warmup，再逐步衰减。warmup 尤其常见于 Transformer 和大模型，因为刚开始参数还不稳定，直接用大学习率容易炸。对推荐系统来说，embedding 和 MLP 往往也会用不同学习率，甚至不同模块单独调。你在面试里如果能说出“学习率是最值得优先调的超参之一”，通常不会错。

可能追问：
- 问：warmup 的作用是什么？答：让模型在训练初期平稳进入有效更新区间，避免大步长导致不稳定。

## 什么是梯度消失和梯度爆炸？
15秒简答：梯度消失是反向传播时梯度越来越小，前层几乎学不到；梯度爆炸是梯度过大导致参数更新失控。

3分钟详答：在深层网络或长序列模型中，梯度需要经过很多层链式传播。如果每层导数都偏小，乘起来就会越来越接近 0，造成梯度消失；如果某些导数偏大，连乘后就会急剧放大，造成梯度爆炸。前者会让模型学不动，后者会导致损失发散、数值溢出。常见缓解方法包括更好的初始化、残差连接、归一化、梯度裁剪和换激活函数。你可以顺带提一句：Transformer 能训练得更深，一个重要原因就是残差和 LayerNorm 大大缓解了这个问题。

可能追问：
- 问：梯度裁剪主要解决哪个问题？答：主要用于缓解梯度爆炸。

## ReLU、GELU、Sigmoid、Tanh 怎么比较？
15秒简答：`ReLU` 简单高效，`GELU` 更平滑且常见于 Transformer，`Sigmoid/Tanh` 易饱和，现代深网隐藏层较少单独使用。

3分钟详答：`Sigmoid` 输出在 0 到 1，适合做概率，但在两端容易饱和，梯度变小；`Tanh` 零中心一点，但同样有饱和问题；`ReLU` 计算简单、梯度稳定，是经典默认选择；`GELU` 可以看成更平滑的门控激活，在 Transformer 和大模型里很常见。推荐排序网络里，隐藏层一般倾向 `ReLU` 或 `GELU`，输出层再根据任务选 sigmoid、softmax 或线性层。面试里如果你能说出“激活函数既影响非线性表达，也影响优化难度”，就不只是背八股了。

可能追问：
- 问：ReLU 的主要问题是什么？答：可能出现神经元长期输出 0，也就是 dying ReLU。

## Softmax 在做什么？
15秒简答：Softmax 把一组实数映射成概率分布，值越大对应概率越高，且所有概率和为 1。

3分钟详答：Softmax 本质是对 logits 做指数化再归一化，所以它会放大相对差异，并把结果限制在 0 到 1 之间。在多分类任务里，这样每个类别就有了可解释的概率。推荐里 softmax 也常用于多候选打分归一化、注意力权重计算和策略网络动作概率输出。你的 PPO 重排里其实就用到了类似思想，把每个候选电影的打分变成概率分布。回答时可以补一句，softmax 前常做数值平移，比如减去最大值，是为了避免指数运算溢出。

可能追问：
- 问：为什么 softmax 前要减最大值？答：为了数值稳定，不改变结果却能避免 `exp` 溢出。

## Attention 里为什么要除以 `sqrt(d_k)`？
15秒简答：因为点积随维度变大容易数值过大，softmax 会过于尖锐，除以 `sqrt(d_k)` 可以稳定梯度和训练。

3分钟详答：自注意力里会计算 `QK^T`，如果键向量维度 `d_k` 很大，点积的方差也会变大，导致 softmax 输入过大，分布非常尖锐，梯度接近 0，训练不稳定。除以 `sqrt(d_k)` 相当于做尺度归一化，让 logits 保持在更合理的范围。这个技巧是 Transformer 论文里的关键细节之一，也属于大模型基础题里很常见的“为什么”。如果你回答时只说“为了稳定”，还不够；最好补上“否则 softmax 饱和、梯度变差”。

可能追问：
- 问：这是经验做法还是有统计意义？答：有统计意义，独立同分布假设下点积方差会随维度增大。

## Transformer Encoder 的核心结构是什么？
15秒简答：核心是“多头自注意力 + 前馈网络 + 残差连接 + LayerNorm”，通过堆叠实现强表达能力。

3分钟详答：Transformer Encoder 可以分成两个大模块。第一个是多头自注意力，它让每个位置都能和序列里其他位置交互，学习全局依赖；第二个是前馈网络，对每个位置做更强的非线性变换。每个模块外面还有残差连接和 LayerNorm，用于稳定训练和保留原信息。多头机制的意义在于让模型从不同子空间关注不同关系，比如语义、位置或行为模式。推荐里的 `SASRec`、`BERT4Rec` 以及大模型本身都建立在这套结构上，所以这是连接推荐和 LLM 的关键基础题。

可能追问：
- 问：残差连接的作用是什么？答：帮助信息和梯度穿透深层网络，缓解训练困难。

## 为什么 Transformer 需要位置编码？
15秒简答：因为自注意力本身对顺序不敏感，位置编码用来告诉模型 token 在序列中的先后关系。

3分钟详答：如果只有自注意力，模型看到的是一堆 token 的集合，不知道谁在前谁在后。可是在语言和序列推荐里，顺序非常重要，比如“先看哪部电影再看哪部电影”会影响兴趣演化。所以 Transformer 需要引入位置编码，把位置信息加到输入表示里。位置编码可以是固定的正余弦，也可以是可学习参数，现代模型还常用旋转位置编码。面试里回答这题时，最好顺便联系推荐序列建模：`SASRec/BERT4Rec` 之所以能建模行为顺序，就是因为既有注意力也有位置信息。

可能追问：
- 问：如果不用位置编码会怎样？答：模型还能学共现关系，但会丢失顺序信息，序列建模能力明显下降。

## 什么是特征交叉？为什么推荐模型喜欢它？
15秒简答：特征交叉就是显式或隐式建模“特征和特征一起出现”的关系，推荐里很多偏好本来就是交互效应。

3分钟详答：推荐系统很少只靠单一特征做决策，更多时候看的是组合关系，比如“年轻用户 + 科幻片 + 周末夜晚”可能比任何一个单特征都更有意义。传统做法会手工构造交叉特征，深度模型则通过 FM、DeepFM、DCN、MLP 等自动学习交互。为什么推荐特别依赖交叉？因为用户偏好常常不是线性的，特征之间存在强耦合。比如同一个导演，在不同类型电影里对不同用户吸引力完全不同。你的项目虽然没有上特别重的工业排序模型，但面试里讲清“推荐的关键之一是学特征交互”，是非常加分的。

可能追问：
- 问：显式交叉和隐式交叉有什么区别？答：显式交叉可解释性更强，隐式交叉更灵活但通常更难解释。

# 推荐系统与经典模型

## 推荐系统一般为什么要分成召回、粗排、精排、重排？
15秒简答：因为全量物品太多，必须分层降复杂度；召回保覆盖，排序保精度，重排再做多样性、公平性和长期价值优化。

3分钟详答：这是推荐面试里出现频率最高的系统题之一。召回层面向海量物品，目标是快速筛出一小批潜在相关候选，所以更看重速度和覆盖；粗排和精排面向越来越小的候选集，模型会更复杂、更注重点击率和转化率；重排则通常不再只追求单点相关性，而是考虑多样性、新颖性、内容生态、长期收益等列表级目标。你的项目虽然规模不大，但结构上已经体现出这套思路：`semantic_hybrid` 像召回候选，`PPO` 更接近重排。面试时如果你能把自己的项目映射到工业架构，效果会比只背定义强很多。

可能追问：
- 问：为什么不能直接一个大模型从全集排序？答：因为计算成本太高、延迟不可控，而且不同阶段优化目标本来就不同。

## 多路召回一般怎么融合？
15秒简答：常见做法有去重拼接、分数归一化后加权融合、规则保底和学习式融合，核心是兼顾覆盖和相关性。

3分钟详答：多路召回存在的意义是“不同召回路有不同偏好和盲区”，比如热门召回偏头部，协同过滤偏行为共现，内容召回偏语义相似，双塔召回偏向学习到的向量邻近。融合时最简单的是各路取前 k 去重后拼接，再按优先级兜底。更进一步会做分数归一化、按路权重融合，或者把召回来源、召回分数、流行度等特征送到后续模型统一学习。你的项目里 `semantic_hybrid + baseline_hybrid + hot` 的回退思路，本质就是规则式融合。面试时可以强调，没有一种召回能吃下全部需求，多路融合是工业常态。

可能追问：
- 问：为什么不同召回路的分数不能直接相加？答：因为分布和尺度可能完全不同，需要先做归一化或校准。

## UserCF 和 ItemCF 的区别是什么？
15秒简答：`UserCF` 找相似用户，`ItemCF` 找相似物品；前者更依赖用户重叠，后者通常更稳定、更适合商品和内容推荐。

3分钟详答：协同过滤是推荐基础题。`UserCF` 的核心是“和你相似的人喜欢什么”，所以要先定义用户相似度，再把相似用户喜欢但目标用户没看过的电影推荐出来；`ItemCF` 则是“你喜欢的电影附近还有什么”，更关注物品之间的共现和相似性。实际工程里，`ItemCF` 往往更稳定，因为物品关系比用户兴趣变化慢，而且物品数通常比用户数更容易维护局部相似表。对你的项目来说，基础协同过滤更像 baseline，方便和后面的语义 embedding、PPO 重排做对照。

可能追问：
- 问：为什么协同过滤会受稀疏性影响？答：因为如果共同交互太少，相似度估计就不可靠。

## 矩阵分解和传统协同过滤相比有什么优势？
15秒简答：矩阵分解把用户和物品映射到共享隐空间里，能在稀疏场景下学到更稳的潜在偏好，比纯相似度法更有泛化能力。

3分钟详答：传统协同过滤依赖显式相似度，容易被稀疏矩阵卡住；矩阵分解则试图把用户-物品交互矩阵分成用户向量和物品向量的乘积，把“相似性”转成潜在空间里的内积关系。这样即使两个用户没有很多完全重叠的电影，也可能通过共享的低维因子建立联系。常见方法包括 SVD、ALS、BPR 等。它的优势是泛化更强、可扩展性更好，但缺点是对冷启动和文本语义不够友好。面试里如果你能说出“矩阵分解解决的是协同信号泛化，不是文本理解”，回答就更完整。

可能追问：
- 问：矩阵分解怎么处理隐式反馈？答：常见做法是把点击/收藏转成偏好强度，并配合置信度或 pairwise loss。

## 双塔召回的基本思想是什么？
15秒简答：双塔把用户和物品分别编码成向量，用向量相似度做快速召回，优点是线上检索效率高，适合大规模召回。

3分钟详答：双塔模型是工业召回的高频题。它会用用户塔编码用户兴趣，用物品塔编码 item 特征，训练目标通常是让正样本用户-物品对更接近、负样本更远。训练好后，物品向量可以离线预计算并建索引，线上只需算用户向量，再做近邻检索。它的优点是速度快、扩展性强，非常适合大规模召回。缺点也很明显：用户和物品在最终打分前交互不充分，所以精细建模能力不如交叉特征很强的排序模型。面试里如果被问“为什么双塔适合召回不适合精排”，这是标准答案。

可能追问：
- 问：双塔为什么在线效率高？答：因为 item 向量可离线缓存，线上只做一次用户编码加 ANN 检索。

## Wide&Deep 解决什么问题？
15秒简答：它同时兼顾“记忆”和“泛化”，Wide 部分学习显式共现规则，Deep 部分学习稠密特征泛化能力。

3分钟详答：Wide&Deep 是推荐系统经典模型。Wide 侧通常是线性模型加人工交叉特征，擅长记住历史上反复出现的强规则；Deep 侧则通过 embedding 和多层感知机学习更高阶、更平滑的泛化能力。为什么它经典？因为推荐里既需要记忆“这个用户经常点这类片”，也需要泛化“虽然没见过这个组合，但从语义上看可能也喜欢”。Wide&Deep 把这两类能力放在一个框架里联合训练，所以非常适合 CTR 和推荐排序。你在项目里没直接实现它也没关系，关键是要知道它代表的是工业排序模型的一个基础范式。

可能追问：
- 问：Wide 一定要手工交叉吗？答：通常是显式交叉或规则特征，优点是可解释、能记忆强规则。

## DeepFM 和 Wide&Deep 的差异在哪里？
15秒简答：DeepFM 用 FM 自动学习低阶交叉，再用 DNN 学高阶交叉，减少了 Wide&Deep 对人工特征工程的依赖。

3分钟详答：DeepFM 可以看作是对 Wide&Deep 的进一步自动化。FM 部分擅长捕捉二阶特征交互，而且不需要人工手写很多交叉；Deep 部分继续负责更高阶的非线性关系。两部分共享同一套 embedding，所以参数更高效。和 Wide&Deep 相比，它的一个主要优点就是弱化了人工构造 wide 特征的工作量，特别适合特征多、组合复杂的场景。面试里通常会问“FM 为什么适合稀疏特征”，你可以补充说，因为它通过向量内积表示交互，比手工枚举所有交叉更高效。

可能追问：
- 问：FM 学到的是几阶交叉？答：标准 FM 主要是二阶交叉，高阶通常交给更深模型处理。

## DCN 为什么会出现？
15秒简答：因为它希望更显式、更高效地学习特征交叉，尤其是有限阶交互关系。

3分钟详答：DCN，也就是 Deep & Cross Network，本质是把显式特征交叉做成网络层。相较于纯 MLP 依赖隐式学习交互，Cross Network 能更直接地构造有限阶组合特征，所以在推荐排序里很受欢迎。它的价值在于：一方面保留了可解释的交叉结构，另一方面比全手工 wide 特征灵活。后续还有 `DCN-V2` 等改进版本，进一步提升表达能力和稳定性。面试里如果你能把 `Wide&Deep、DeepFM、DCN` 讲成“显式交叉和隐式交叉的不同路线”，会显得结构非常清晰。

可能追问：
- 问：DCN 比纯 MLP 的优势是什么？答：更擅长显式学习有限阶交叉，样本效率和可解释性通常更好。

## DIN 的核心思想是什么？
15秒简答：DIN 的核心是“不同候选商品应激活用户历史中的不同兴趣”，所以它用 attention 做 target-aware 兴趣提取。

3分钟详答：DIN 是中国互联网推荐面试几乎必考模型。传统做法往往把用户历史行为平均池化成一个固定向量，但这会把“用户喜欢什么”平均掉。DIN 认为用户兴趣是多样的，而且要看当前候选 item 是什么。比如用户同时喜欢科幻和爱情片，当候选是《星际穿越》时，历史里的科幻片应该被更强激活。于是 DIN 用 attention 让候选 item 和历史行为做交互，得到一个 target-aware 用户表示。它最大的价值在于把“兴趣不是静态的”这件事显式建模出来。面试里能把这点说清楚，基本就答到位了。

可能追问：
- 问：DIN 解决了平均池化的什么问题？答：解决了把不同兴趣混成一个固定向量、缺少候选感知的问题。

## DIEN 或 SIM 这类模型相比 DIN 又往前走了什么？
15秒简答：它们进一步建模了兴趣的动态演化和长序列问题，不只是“激活兴趣”，还关注兴趣如何随时间变化。

3分钟详答：DIN 已经能做 target-aware attention，但它对用户兴趣的时间演化考虑还不够。DIEN 在此基础上引入兴趣抽取层和兴趣演化层，通常会用 GRU 一类结构建模行为序列，强调“用户兴趣不是静止向量，而是随时间变化的状态”。而 SIM 等模型会更关注超长行为序列的高效检索和匹配，解决长序列场景下算力和注意力稀释问题。面试里如果被问到从 DIN 到 DIEN 的演化，你可以总结成“从静态兴趣激活，走向动态兴趣建模”。

可能追问：
- 问：为什么长行为序列是个难点？答：因为噪声多、计算贵，而且早期行为和当前兴趣未必同等重要。

## SASRec 的核心思想是什么？
15秒简答：SASRec 用自注意力做序列推荐，重点建模用户最近行为和远距离依赖之间的关系。

3分钟详答：SASRec 是序列推荐里非常经典的 Transformer 路线。和 RNN 相比，自注意力更容易并行，也更擅长捕捉长距离依赖。SASRec 的直觉是：用户下一个想看的电影，往往和最近若干次行为强相关，但有时更久远的兴趣也有价值。自注意力可以自动分配不同历史行为的权重，而不需要像传统序列模型那样一步步递推。它特别适合行为顺序明显的场景，比如电商、视频、音乐和电影推荐。面试里如果你能把 SASRec 说成“推荐领域把 Transformer 用到行为序列上的代表模型”，通常就够用了。

可能追问：
- 问：SASRec 和 DIN 的核心差异是什么？答：DIN 是 target-aware 激活历史兴趣，SASRec 更强调用自注意力直接建模行为序列关系。

## BERT4Rec 和 SASRec 有什么不同？
15秒简答：SASRec 更偏单向序列预测，BERT4Rec 用双向 Transformer 和 mask 任务建模序列上下文，表达更强但训练方式不同。

3分钟详答：BERT4Rec 把 NLP 里的 BERT 思路带到推荐序列建模里。它不再只根据过去预测未来，而是通过随机 mask 掉序列中的物品，让模型利用双向上下文去恢复被遮住的 item。这样做的好处是能学到更充分的序列表示，但和严格的自回归预测相比，训练目标会更像表示学习。SASRec 更接近按时间顺序做下一物品预测。两者没有绝对谁碾压谁，更多看数据规模、序列长度和任务目标。面试里能说出“BERT4Rec 是双向掩码建模路线”，通常就已经超过基础线了。

可能追问：
- 问：双向建模的好处是什么？答：能利用更完整的上下文信息学习序列表示。

## 多目标学习为什么在推荐里重要？
15秒简答：因为真实业务不只关心点击，还关心转化、停留时长、留存等，单目标模型容易把系统优化偏。

3分钟详答：推荐系统如果只追求 CTR，往往会变成“标题党”和短期刺激优化器”，但业务真正关心的可能还有播放完成率、收藏、购买、留存甚至长期价值。因此工业系统常用多目标学习，把多个相关任务一起建模。典型结构有 `ESMM` 解决点击和转化链路问题，`MMoE/PLE` 解决多目标之间共享与冲突问题。面试里不一定要求你把每个结构都讲很细，但至少要知道：多目标学习的本质是利用共享信息，同时避免某个目标独占模型能力。你的项目里虽然主要是推荐满意度优化，但 PPO 奖励里其实已经隐含了多目标权衡。

可能追问：
- 问：多目标一定比单目标好吗？答：不一定，如果目标冲突严重或样本质量不一致，反而可能互相干扰。

## 推荐系统怎么做冷启动？
15秒简答：用户冷启动靠热门、人群先验和内容理解，物品冷启动靠文本、图像、类目等内容特征，核心是减少对交互数据的依赖。

3分钟详答：冷启动分用户冷启动和物品冷启动。用户冷启动时没有行为序列，常见方案是热门榜、品类精选、问卷、注册信息、人群画像和上下文特征；物品冷启动时没有交互反馈，常依赖标题、简介、类目、图像、作者信息等内容特征。对你的电影项目来说，电影本身有丰富文本元数据，所以物品冷启动相对好做；用户冷启动则主要依赖热门片和内容型推荐。面试里最好别把冷启动只回答成“推热门”，那样太浅。要强调“内容特征和人群先验是关键补偿信息”。

可能追问：
- 问：冷启动为什么是推荐的根问题之一？答：因为大多数协同信号都依赖历史交互，而新用户新物品恰好没有这些数据。

## 什么是长尾问题？推荐系统为什么容易头部化？
15秒简答：长尾问题是大部分内容交互很少，推荐系统如果只追求短期点击，容易把流量持续集中到头部内容上。

3分钟详答：推荐系统天然会放大马太效应。头部物品有更多曝光和反馈，所以模型更容易学到它们，并继续推荐它们；长尾内容因为缺少历史数据，很难进入候选和排序前列。这会带来两个问题：一是用户看到的内容越来越同质，二是平台生态越来越差。解决方法包括探索机制、覆盖率约束、多样性重排、长尾召回通道和内容理解增强。你的 PPO 重排里 coverage bonus 就是在解决这个问题。面试里如果你能把“长尾不是单纯没流量，而是系统性反馈闭环导致的”说出来，会很加分。

可能追问：
- 问：一味扶持长尾有风险吗？答：有，可能牺牲相关性和短期指标，所以需要平衡约束而不是强行平均曝光。

## 曝光未点击能直接当负样本吗？
15秒简答：通常比“未曝光未点击”更适合作负样本，但仍要考虑曝光位置、吸引力偏差和展示质量偏差。

3分钟详答：很多面试官爱问这题，是因为它正好考推荐里的因果和偏差意识。曝光未点击比“完全没见过”更接近真实负反馈，所以通常更适合当负样本；但它也不等于纯粹不喜欢，因为没点可能是位置太靠后、封面不吸引、上下文不合适、用户当时没空看。所以工业里会对曝光位置、时段、上下文做校正，或者引入反事实学习、IPS 一类方法缓解偏差。你的项目没有做复杂去偏也没关系，至少要说出：曝光未点击是相对更好的负例，但不是完美真负样本。

可能追问：
- 问：为什么未曝光样本不能随便当负例？答：因为用户根本没机会看到，不能说明他不喜欢。

## 推荐里常见的偏差有哪些？
15秒简答：常见有曝光偏差、位置偏差、选择偏差、流行度偏差和反馈回路偏差。

3分钟详答：推荐系统不是在理想数据上学习，而是在“系统已经做过一次选择”的日志上学习，所以偏差很多。曝光偏差是某些内容从来没机会被看见；位置偏差是越靠前越容易被点；流行度偏差是热门内容因为已有反馈更多而更容易继续被推；反馈回路偏差是系统推荐什么、就更容易收集到什么数据。这些偏差会让模型把“被展示”误当成“天然更好”。面试里如果你能说出偏差不仅是统计问题，也是系统问题，回答会很成熟。你的项目里通过覆盖率奖励和探索机制，实际上就在缓解部分偏差积累。

可能追问：
- 问：位置偏差最直接的影响是什么？答：模型可能学到“位置高就更容易点”，而不是内容本身更相关。

## 离线评估和线上 A/B 测试的区别是什么？
15秒简答：离线评估快、便宜、便于迭代，但不能完全代表真实用户行为；线上 A/B 最真实，但成本高、风险也高。

3分钟详答：离线评估用历史数据回放模型效果，适合快速做算法筛选和消融实验；线上 A/B 直接面对真实用户流量，能观察点击、停留、转化、留存等真实业务指标，但有流量成本和回滚风险。二者不是替代关系，而是递进关系。通常先用离线评估排掉明显不行的方案，再让少数候选进入 A/B。你的项目因为是本地闭环，没有真实大流量，所以重点在离线对比和实验快照，但你要知道工业上最终还是要看线上结果。这个态度很重要。

可能追问：
- 问：为什么离线高分模型线上不一定好？答：因为日志偏差、目标不一致和用户行为分布变化都会造成离在线不一致。

## 树模型和深度模型在推荐里怎么选？
15秒简答：树模型适合中小数据、强表格特征、解释性要求高的场景；深度模型适合高维稀疏特征、embedding 和复杂交互学习。

3分钟详答：这题近几年在实习面试里也很常见。像 `XGBoost/LightGBM` 这类树模型在结构化表格数据上很强，调起来相对稳，特征重要性也更容易解释；但它对大规模稀疏 id、embedding 联合学习、多模态输入和序列行为建模就没那么自然。深度模型虽然调参更复杂，但更适合用户 id、item id、文本向量、序列行为一起建模。推荐工业里经常是召回、排序和重排用不同模型，甚至树和深度混用。你可以回答：如果是小数据、特征工整的快速 baseline，我会先上树模型；如果要做用户-物品交互、序列和多模态，深度模型更合适。

可能追问：
- 问：树模型能处理 embedding 吗？答：可以把 embedding 当连续特征输入，但它不擅长端到端学习 embedding。

## 你的项目为什么没有一开始就上双塔、DIN、BERT4Rec？
15秒简答：因为当前项目首先要解决的是“轻量、可运行、可解释”的闭环问题，先用 baseline、语义 embedding 和 PPO 把核心思路跑通更合适。

3分钟详答：面试官有时会故意问“你为什么不用更高级的模型”。这时候不要慌，重点是说明约束和阶段目标。你的项目是单机实验系统，数据规模、特征规模和算力都有限。如果一开始就上双塔、DIN 或 BERT4Rec，工程复杂度会明显升高，效果却不一定稳定。更重要的是，小项目最需要的是可对比的 baseline 和清晰的优化路径，而不是一堆很难解释的模型名。你可以说：我知道这些模型，也知道它们各自适合什么阶段；当前项目先解决候选质量、多样性和闭环训练，后续如果做工业级演进，我会优先考虑双塔召回和序列排序。

可能追问：
- 问：如果只允许你升级一个模型，你优先升级哪里？答：我会优先升级召回，用双塔或序列召回提升候选质量，再决定排序层是否同步升级。

# 强化学习与推荐重排

## 强化学习的四要素是什么？
15秒简答：状态、动作、奖励、环境，这是描述 RL 问题的最基本四个组成部分。

3分钟详答：强化学习讲的是智能体在环境中根据状态选择动作，收到奖励后继续更新策略的过程。状态表示当前可观测信息，动作表示当前可做决策，奖励表示这次决策好不好，环境则负责给出状态转移和反馈。推荐场景里，状态可以是用户画像和上下文，动作可以是推荐哪个 item 或列表，奖励可以是点击、观看、停留、购买等，环境就是用户与系统的交互过程。面试里如果你能立刻把这四要素映射到推荐系统，答案就很完整。

可能追问：
- 问：RL 和监督学习最大的不同是什么？答：RL 要考虑动作影响未来状态和长期回报，不只是拟合固定标签。

## 怎么把推荐系统建模成 MDP？
15秒简答：把用户当前画像和上下文当状态，把推荐决策当动作，把用户反馈当奖励，把用户行为变化看成状态转移，就可以近似建成 MDP。

3分钟详答：推荐系统并不是天然严格满足马尔可夫性，但常常会近似为 MDP。比如用户当前兴趣、最近行为、会话上下文可以组成状态；系统推荐的电影或推荐列表是动作；点击、收藏、评分、观看时长构成奖励；用户在看到推荐后兴趣变化、继续浏览下一页就是状态转移。这样建模的价值是可以把推荐优化从“预测当前点不点”提升到“优化长期用户价值”。当然，真实推荐里环境是部分可观测、非平稳的，所以这是近似建模，不是完美理论模型。面试里主动提这点会显得更稳。

可能追问：
- 问：推荐一定满足马尔可夫性吗？答：通常只是近似满足，真实用户偏好有隐藏状态，所以更像 POMDP。

## Multi-Armed Bandit 和完整 RL 有什么区别？
15秒简答：Bandit 只关心当前动作的即时奖励，不建模长期状态转移；完整 RL 要考虑动作对未来状态和长期收益的影响。

3分钟详答：Bandit 可以看作 RL 的简化版。它每次面对一个上下文，选择一个动作，立即获得奖励，但默认当前动作不会影响未来环境结构，所以没有复杂的状态转移问题。这非常适合新闻推荐、流量分配、探索利用等场景。完整 RL 则更复杂，因为当前推荐会影响用户未来兴趣和未来可收集到的数据，所以要优化累计回报。面试里如果被问“推荐里为什么很多时候 bandit 就够了”，答案可以是：当决策周期短、未来影响弱或建模成本太高时，bandit 更实用；当你明确关心长期留存和序列效应时，才更有必要上 RL。

可能追问：
- 问：探索利用问题更典型出现在 bandit 还是 RL？答：两者都有，但 bandit 是最经典、最直接的场景。

## On-policy 和 Off-policy 有什么区别？
15秒简答：On-policy 用当前策略生成的数据学当前策略，Off-policy 可以用其他策略收集的数据学习目标策略。

3分钟详答：这是 RL 基础高频题。On-policy 的代表是 PPO，它通常要求数据来自当前或接近当前的策略，所以训练稳定但样本利用率相对低。Off-policy 的代表是 DQN、DDPG 一类，可以利用经验回放中的旧数据甚至别的策略生成的数据，因此样本效率更高，但训练容易不稳定。推荐场景里因为真实用户交互成本高，大家都很看重样本效率，所以 off-policy 和离线 RL 常被讨论；但在小项目里，PPO 这类 on-policy / 近 on-policy 方法因为实现简单、稳定性更好，也很适合展示思路。

可能追问：
- 问：为什么 off-policy 样本效率更高？答：因为同一批历史数据可以反复利用，不必每次都重新采样。

## Q-learning 和 DQN 的关系是什么？
15秒简答：Q-learning 是经典值函数方法，DQN 用神经网络近似 Q 函数，把它扩展到更大状态空间。

3分钟详答：Q-learning 的核心思想是学习状态-动作价值 `Q(s,a)`，每次选价值最大的动作。它在小状态空间里可以用表格法实现，但一旦状态复杂，比如图像、长序列或高维特征，就没法直接查表了。DQN 就是用深度网络来近似 Q 函数，同时引入经验回放和 target network 提升训练稳定性。面试里如果被问为什么推荐里不用太多 DQN，可以回答：因为动作空间经常是动态候选集、列表级动作，而且点值函数方法不一定最适合序列重排，所以很多场景更偏向策略梯度或 actor-critic。

可能追问：
- 问：经验回放解决什么问题？答：打破样本相关性，提高数据利用率和训练稳定性。

## 什么是策略梯度？
15秒简答：策略梯度直接对参数化策略求梯度，目标是提高高回报动作的概率、降低低回报动作的概率。

3分钟详答：和 Q-learning 先学价值再选动作不同，策略梯度是直接优化策略本身。它把策略写成一个参数化分布，比如给每个候选电影分配概率，然后通过采样得到回报，再根据回报大小调整参数。它的优点是天然适合连续动作或随机策略，也更容易处理复杂约束；缺点是方差大、训练不稳定。推荐里如果你要做列表构造、探索策略或组合优化，策略梯度往往比简单的值函数方法更顺手。你的 PPO 本质上就是策略梯度家族的改进版。

可能追问：
- 问：策略梯度为什么容易高方差？答：因为它依赖采样回报，奖励噪声和环境随机性都会直接进入梯度估计。

## Actor-Critic 的基本思想是什么？
15秒简答：Actor 负责出动作策略，Critic 负责评估当前状态或动作好不好，两者配合降低纯策略梯度的方差。

3分钟详答：纯策略梯度虽然直接，但方差大；纯值函数方法虽然稳定一些，但在复杂策略空间里不够灵活。Actor-Critic 就是把两者结合起来。Actor 输出动作分布，Critic 估计价值函数或优势函数，用来告诉 Actor 这次动作相对于平均水平是好还是坏。这样既能保持策略优化的灵活性，又能减小更新噪声。PPO、A2C、A3C 等都属于这条路线。面试里说清楚“Critic 不负责最终执行动作，而是提供学习信号”，是一个不错的细节点。

可能追问：
- 问：优势函数为什么有用？答：它看的是相对好坏而不是绝对回报，更利于稳定更新。

## PPO 的核心思想是什么？
15秒简答：PPO 通过限制策略每次更新幅度，避免新旧策略差太远，从而在稳定性和实现复杂度之间取得平衡。

3分钟详答：PPO 是近几年 RL 面试最常见的算法之一。它继承了策略梯度和 actor-critic 的思想，但重点解决的是“策略一下子更新太猛，训练容易崩”的问题。PPO 会比较新旧策略在同一动作上的概率比值，如果变化过大，就通过 clip 机制限制目标函数收益，防止策略一步跨太远。这样做不像 TRPO 那样需要复杂约束优化，实现更简单，却能保持不错的训练稳定性。你的项目选择 PPO 做重排很合理，因为它在小规模策略优化上足够轻量，也方便解释。

可能追问：
- 问：PPO 为什么比原始 policy gradient 更稳？答：因为它限制了更新步长，避免新策略和旧策略差异过大。

## PPO 的 clip objective 在做什么？
15秒简答：它用概率比值裁剪过大的更新，让策略即使在高优势样本上也不会被一步推得过头。

3分钟详答：PPO 的目标函数里会出现新旧策略概率比值 `ratio`。如果某个动作回报很好，模型本来会想大幅提高它的概率；如果回报很差，就想大幅降低概率。但这种大幅更新会让训练不稳定。所以 PPO 用 `clip` 把 ratio 限制在一个区间，比如 `[1-eps, 1+eps]`。当更新超过区间后，就不再继续放大收益。这样本质上是在做保守更新。面试里如果能把这点讲成“限制策略漂移”，通常就够用了。你项目代码里的 `PPO_CLIP_EPSILON` 就对应这个思想。

可能追问：
- 问：clip 太小或太大会怎样？答：太小学得慢，太大又失去稳定更新的意义。

## 什么是 Advantage 和 GAE？
15秒简答：Advantage 表示某动作相对平均水平好多少，GAE 是更平滑的优势估计方法，用来平衡偏差和方差。

3分钟详答：如果只用总回报更新策略，方差往往很大，所以会引入 baseline，比如状态价值函数，得到 `A(s,a)=Q(s,a)-V(s)`。它表示在当前状态下，这个动作比平均水平到底好多少。GAE，也就是 Generalized Advantage Estimation，是一种把多步 TD 误差加权累积起来的优势估计方法，可以在偏差和方差之间做平衡。虽然你的小项目实现里没有完整展开 GAE 公式，但在面试里知道它的作用很重要。因为很多面试官问 PPO，不是真的让你推公式，而是看你是否理解为什么要估计 advantage。

可能追问：
- 问：为什么 advantage 比直接用 reward 更好？答：因为它减去了状态基线，能显著降低梯度方差。

## 为什么 PPO 比较适合做推荐重排？
15秒简答：因为重排动作空间相对小，且目标不只相关性，还包括多样性和长期收益，这正是 PPO 这类策略优化方法擅长的场景。

3分钟详答：推荐重排不像召回那样面对海量物品，而是在有限候选里做顺序决策。这时 RL 的两个优势会更明显。第一，可以把覆盖率、多样性、探索等非单点相关性目标写进奖励里；第二，可以考虑短期满意度和长期反馈之间的权衡。PPO 又比很多 RL 算法实现更轻、更稳，所以很适合作为项目级重排方案。你在面试里可以说：我没有夸大 PPO 的能力，而是把它放在最合适的层，即小候选集上的策略优化。这样的回答通常很讨喜。

可能追问：
- 问：如果候选集只有 10 个 item，还值得用 PPO 吗？答：如果只追求单点点击未必值得，但如果要优化列表级目标，仍然有意义。

## 推荐里的探索与利用怎么理解？
15秒简答：利用是推当前最有把握的内容，探索是给不确定但可能好的内容一些曝光机会，核心是短期收益和长期学习之间的平衡。

3分钟详答：如果系统永远只推当前最优估计的内容，会越来越保守，历史上没被充分曝光的电影永远没有机会；但如果探索过多，又会伤害当前用户体验。所以推荐系统需要在探索和利用之间找平衡。经典方法包括 `ε-greedy`、UCB、Thompson Sampling、bandit 策略和 RL 策略约束。你的项目在 PPO 重排里用小比例 `ε-greedy`，就是一种轻量探索机制。面试里最好能明确说出：探索不是随机乱推，而是为了收集更有价值的反馈，改善长期学习。

可能追问：
- 问：为什么推荐系统会天然过度利用？答：因为已有高反馈内容会持续拿到更多曝光，形成自增强闭环。

## 你项目里的 `ε-greedy` 和 PPO 怎么配合？
15秒简答：PPO 给出主策略分数，`ε-greedy` 在小概率下偏向探索低曝光候选，两者结合能兼顾主策略效果和长尾探索。

3分钟详答：这题很适合从代码出发讲实现。我的 PPO 会先根据用户状态和候选电影特征算出策略分数，再叠加 coverage bonus 得到初始 `rl_score`。随后在构造最终列表时，大部分情况下按综合分数选最优候选；少部分情况下触发 `ε-greedy`，按覆盖率相关权重随机选。这样设计不是替代 PPO，而是给 PPO 一个受控探索出口。否则如果完全按当前策略贪心选，模型长期可能只在一小撮头部电影附近打转。这个回答很能体现你是理解实现细节的。

可能追问：
- 问：为什么探索时按覆盖率加权而不是完全均匀随机？答：因为我们希望更有针对性地补偿低曝光内容，而不是无差别随机。

## 什么是 reward shaping？它的风险是什么？
15秒简答：Reward shaping 是在主奖励外加入辅助奖励帮助学习，但如果设计不当，模型会“钻奖励漏洞”而偏离真实目标。

3分钟详答：真实业务奖励往往稀疏又延迟，比如用户是否长期留存、是否真正满意，很难直接优化。所以大家会人为加入一些辅助奖励，比如点击奖励、探索奖励、多样性奖励、覆盖率奖励等，这就是 reward shaping。它的价值是让训练更稳定、目标更可学；风险是模型可能学会优化代理目标而不是真实目标。比如你如果把多样性权重拉太高，系统可能为了“看起来不一样”而牺牲相关性。面试里主动提到“reward hacking”或“代理目标偏移”，会让答案更扎实。

可能追问：
- 问：怎么降低 reward shaping 风险？答：让辅助奖励权重可控，并用离线和线上指标一起校验是否偏离主目标。

## 稀疏奖励为什么是 RL 推荐的难点？
15秒简答：因为用户真实强反馈少、延迟长，导致策略很难从有限信号里稳定学习出有效行为。

3分钟详答：推荐系统里真正高价值的信号往往很稀疏，比如高评分、完播、购买、留存，这些都不像点击那样高频。结果是 RL 每做一次动作，很可能拿到的反馈很弱、很噪甚至延迟很久，这会显著提高学习难度。常见缓解思路包括把强弱反馈分层设计、引入辅助奖励、做短期代理目标、用模拟器预训练或离线日志先 warm start。你的项目里对评分、收藏、点击分别给不同强度奖励，本质上就是在缓解稀疏奖励问题。

可能追问：
- 问：点击算不算稀疏奖励的缓解？答：算，它是高频弱反馈，常作为强反馈缺失时的代理信号。

## 什么是离线 RL？它为什么难？
15秒简答：离线 RL 只用历史日志学习策略，不在线探索；难点在于分布外动作估计不可靠、偏差容易被放大。

3分钟详答：离线 RL 看起来很适合推荐，因为线上探索成本高，大家都想直接用历史日志学策略。但问题是，历史日志只覆盖了旧策略实际做过的动作。如果新策略想选一些日志中几乎没出现过的动作，模型就必须对“分布外动作”做外推，这非常不可靠。于是 Q 值容易被高估，策略会学到看似很优但真实表现很差的行为。推荐面试里如果问离线 RL，最重要的是说出 `distribution shift` 和 `extrapolation error`。你的项目更偏在线反馈入池加批量训练，不是严格离线 RL，但知道这个概念会很加分。

可能追问：
- 问：为什么离线 RL 容易高估没见过的动作？答：因为模型只能靠函数逼近外推，而不是靠真实反馈验证。

## 什么是 slate recommendation？它为什么更难？
15秒简答：slate recommendation 是一次性推荐一个列表而不是单个 item，难点在于列表内部 item 会相互影响，奖励也常是整体性的。

3分钟详答：很多真实推荐不是一次只给一个内容，而是给一屏列表。这时问题不再是“哪个 item 最好”，而是“哪组 item 按什么顺序最好”。列表里电影之间会相互竞争注意力，也可能彼此补充或重复，所以奖励不再是各个 item 分数简单相加。重排里的多样性、公平性和展示顺序都属于 slate 问题。你的 PPO 重排虽然实现上更接近逐个选择候选，但优化目标本质上是列表体验，这就已经很接近 slate recommendation 了。面试里把这一点说出来，会显得你对问题抽象有一定高度。

可能追问：
- 问：为什么列表级目标比 item 级目标更难？答：因为 item 之间存在组合效应，目标函数不再可简单分解。

## 推荐系统为什么会用模拟器，比如 RecSim？
15秒简答：因为真实线上探索成本高、风险大，模拟器可以先验证策略方向、做离线实验和生成可控交互数据。

3分钟详答：RL 推荐最大的现实难点是没有安全、低成本的环境可以反复试错。模拟器的价值就在这里。像 `RecSim` 这样的工具会模拟用户兴趣、推荐交互和反馈过程，让研究者先在可控环境里比较策略。它不能完全替代真实线上数据，但非常适合做算法原型验证、参数敏感性分析和教学实验。面试里如果被问“为什么你的项目没有模拟器”，你可以说当前数据规模不大、重点是跑真实闭环；但如果未来做更系统的 RL 推荐研究，引入模拟器会是很合理的下一步。

可能追问：
- 问：模拟器最大的局限是什么？答：环境假设可能和真实用户差很远，策略容易在模拟器里过拟合。

## PPO、DPO、GRPO 有什么区别？
15秒简答：PPO 是通用 RL 策略优化；DPO 更像把偏好学习转成监督式优化；GRPO 是近年大模型 RL 里用于组内相对奖励优化的变体。

3分钟详答：这题近两年在大模型和 RL 面试里都很常见。PPO 来自传统强化学习，依赖策略梯度和价值估计，强调稳定更新；DPO 主要用于偏好对齐，它不显式训练 reward model 和 value model，而是直接用偏好对比较优化策略，更像“有偏好标签的对数比率学习”；GRPO 则是一些大模型后训练里使用的组相对优化思路，用样本组内相对表现来更新策略。对推荐系统岗位来说，不一定要求你深入推导，但至少要知道它们的使用场景不同。你的项目采用 PPO，是因为这里问题是推荐重排策略优化，不是大模型人类偏好对齐。

可能追问：
- 问：DPO 为什么常被说比 PPO 简单？答：因为它绕开了显式 reward model 和 value model 的复杂训练流程。

## 为什么 RL 不一定比监督学习排序更好？
15秒简答：因为 RL 训练更难、样本效率更低、调参更复杂，如果业务目标主要是短期相关性，监督排序往往更稳更划算。

3分钟详答：这其实是非常成熟的一道题。很多同学会把 RL 说成“更高级所以更好”，但真实工业里不是这样。监督学习排序直接用大量点击日志学当前相关性，训练稳定、样本多、评估也成熟；RL 只有在你明确关心长期价值、探索利用、列表级约束或多步决策时，优势才会更明显。而且 RL 对环境建模、奖励设计和分布偏移都很敏感。你的项目把 PPO 放在重排层，而不是把它说成完全替代监督排序，本身就是一种很合理的工程判断。

可能追问：
- 问：什么时候你会优先不用 RL？答：当目标主要是短期点击预测、数据反馈稳定且列表级约束不强时，我会优先用监督排序。

# 大模型基础与 LLM4Rec

## Transformer 为什么成为大模型基础架构？
15秒简答：因为它用自注意力高效建模长距离依赖，易并行扩展，适合大规模预训练。

3分钟详答：Transformer 取代 RNN/LSTM 的关键原因有三点。第一，自注意力能直接建模任意位置间关系，不像 RNN 那样必须逐步递推；第二，训练时并行度高，非常适合大规模 GPU 集群；第三，随着层数、参数量和数据量增大，Transformer 展现出了很强的扩展性。大模型之所以能在语言、代码、多模态等任务上统一，很大程度上都建立在这套结构之上。面试里最好别只说“因为 attention 很强”，还要补上“它特别适合大规模训练和扩展”。

可能追问：
- 问：Transformer 的主要缺点是什么？答：标准自注意力复杂度高，长序列时算力和显存成本都很大。

## Q、K、V 分别是什么？
15秒简答：Q 是当前查询，K 是可匹配的键，V 是匹配后要取出的值；attention 本质上是在用 Q 和 K 算权重，再加权汇总 V。

3分钟详答：这是 Attention 最经典的解释题。可以把它类比成数据库检索或信息检索。每个 token 都会映射成 Query、Key、Value 三个向量。当前 token 的 Query 和所有 token 的 Key 做相似度计算，得到关注权重；然后用这些权重对对应的 Value 加权求和，得到新的上下文表示。之所以分成 Q、K、V，而不是一个向量打天下，是为了让“匹配关系”和“传递内容”有不同的投影空间。面试里如果你能用“谁在找什么、根据什么匹配、最后拿到什么信息”来解释，通常很清楚。

可能追问：
- 问：为什么不直接用原始向量做 attention？答：因为不同投影空间能让模型更灵活地学匹配关系和内容表达。

## 自注意力的复杂度为什么是 `O(n^2)`？
15秒简答：因为长度为 `n` 的序列里，每个位置都要和其他所有位置算一次相似度，所以两两交互是平方级。

3分钟详答：标准 self-attention 要构造一个 `n x n` 的注意力矩阵，其中每个元素表示两个位置之间的相关性。于是时间和空间复杂度都大致随 `n^2` 增长。这也是为什么长文本、超长行为序列和长上下文大模型都在研究稀疏注意力、线性注意力和 chunk attention。推荐里的序列模型如果历史过长，也会遇到类似问题，所以才会有 SIM 这类强调长序列高效建模的工作。面试里答到这里，通常就足够了。

可能追问：
- 问：为什么长上下文模型很贵？答：因为注意力矩阵和 KV cache 都会随着序列长度快速膨胀。

## 预训练、SFT、RLHF 分别是什么？
15秒简答：预训练学通用语言能力，SFT 用高质量指令数据做监督微调，RLHF 再根据偏好或奖励信号对齐模型输出。

3分钟详答：大模型训练通常是分阶段的。预训练阶段用海量文本做下一词预测，学到语言知识和通用表示；SFT，即监督微调，用高质量问答、任务数据让模型学会按照指令回答；RLHF 则进一步引入人类偏好或奖励信号，让模型更符合“有帮助、无害、诚实”的目标。近年来也有 DPO 这类更简化的偏好优化路线。面试里如果你能说出三者分工，通常就能应付大部分大模型基础问题。对推荐系统岗位来说，这套知识有助于理解 LLM4Rec 里为什么很多方案更偏 SFT 或 instruction tuning，而不是直接 RL。

可能追问：
- 问：SFT 和预训练最大的区别是什么？答：预训练学通用语言分布，SFT 学具体任务格式和对话行为。

## LoRA 和 QLoRA 的核心思想是什么？
15秒简答：LoRA 通过低秩增量矩阵微调大模型，减少可训练参数；QLoRA 则进一步把底座量化，降低显存占用。

3分钟详答：全量微调大模型代价太高，所以 LoRA 提出只在部分线性层旁边插入低秩矩阵，冻结原模型权重，只训练这部分小参数。这样可以显著降低显存和训练成本。QLoRA 则更进一步，把底座模型以更低比特量化存储，再配合 LoRA 微调，进一步压缩资源消耗。面试里如果你在讲 LLM4Rec 或推荐文本增强方案，LoRA 是非常高频的基础题。你可以补一句，LoRA 适合快速适配领域任务，但当任务和底座分布差异很大时，也未必总是足够。

可能追问：
- 问：LoRA 为什么能工作？答：因为很多任务适配所需的权重更新本身具有低秩结构。

## 什么是 RAG？它和微调有什么区别？
15秒简答：RAG 是“检索增强生成”，通过外部知识检索补充上下文；微调是把知识或行为模式写进参数。

3分钟详答：RAG 常用于把模型知识与外部文档库连接起来。流程通常是把文档编码存库，用户提问时先检索相关片段，再把检索结果连同问题一起喂给大模型生成答案。这样更新知识不必重新训练模型，也更容易给出处。微调则是通过训练修改模型参数，让模型内化某类任务行为或领域知识。二者不是对立关系，而是解决不同问题。对于推荐系统，RAG 可用于结合电影资料库、用户评论或知识图谱给出解释和问答；微调更适合让模型学会推荐领域特定格式和策略。

可能追问：
- 问：RAG 为什么不能彻底替代微调？答：因为检索能补知识，但未必能学到稳定的任务行为和领域决策风格。

## 大模型幻觉是什么？怎么缓解？
15秒简答：幻觉是模型生成了看起来流畅但不真实的内容；常见缓解方式包括 RAG、约束输出、工具调用和更严格的评估。

3分钟详答：幻觉的根源在于语言模型本质上是在预测最可能的下一个 token，而不是内建事实校验器。所以当知识缺失、上下文不足或问题本身含糊时，模型可能编造细节。缓解方法通常包括：用 RAG 提供可追溯证据；让模型输出结构化结果；加入拒答策略；做工具调用验证；在高风险场景里设置规则或人工审核。推荐场景里如果让大模型做“推荐理由生成”或“对话式推荐”，就要尤其注意别让它编造电影信息。面试里知道“幻觉不是完全消除，而是控制风险”，回答会更成熟。

可能追问：
- 问：温度调低能完全解决幻觉吗？答：不能，只能减少随机性，知识缺口和推理错误依然可能存在。

## 常见 decoding 策略有哪些？
15秒简答：常见有 greedy、beam search、top-k、top-p 和 temperature sampling，不同策略在稳定性和多样性之间做权衡。

3分钟详答：解码策略决定了模型如何从下一个 token 分布中选词。Greedy 最稳定，但容易保守；beam search 更偏寻找高概率整体序列，但在开放式生成里未必总自然；top-k 和 top-p 通过截断候选空间保留随机性，更适合创意和对话场景；temperature 用来调节分布平滑程度。推荐问答场景通常希望答案稳定可信，因此温度往往设低一些。面试里如果被问“为什么推荐理由生成不宜高温度”，你可以直接说：因为我们更看重一致性和事实性，而不是文学创造力。

可能追问：
- 问：top-p 和 top-k 的区别是什么？答：top-k 固定候选数，top-p 固定累计概率阈值，更自适应。

## Embedding Model 和 Reranker 有什么区别？
15秒简答：Embedding 模型擅长高效召回，Reranker 擅长精细判断相关性；前者快，后者准，但更贵。

3分钟详答：这是检索和推荐融合场景里的常见题。Embedding 模型把文本、用户描述或 item 描述编码成向量，适合用 ANN 做大规模近邻检索，所以特别适合召回。Reranker 通常会对 query 和候选做更充分的交互式编码，比如 cross-encoder，因此相关性判断更准确，但计算成本高，不适合全量检索。推荐系统里这和“召回-排序”分层思想高度一致。放到 LLM4Rec 里，可以把大模型 embedding 用在语义召回，把更强但更慢的模型放在小候选集精排。

可能追问：
- 问：为什么 reranker 不能直接拿来全量召回？答：因为它需要对 query 和每个候选做交互计算，延迟和成本都太高。

## Prompt tuning、LoRA、全量微调怎么选？
15秒简答：Prompt tuning 成本最低但能力有限，LoRA 是常见折中，全量微调最强但最贵，选择取决于资源和任务差异。

3分钟详答：如果任务只是格式引导或轻度风格适配，prompt 设计甚至 prompt tuning 就可能够用；如果需要较强的领域适配但资源有限，LoRA/QLoRA 是性价比最高的选择；如果任务和底座差异巨大、且资源充足，才会考虑全量微调。面试里最忌讳把答案说成“全量微调最好”，因为工程上往往不可接受。对推荐系统来说，很多场景其实不需要改到底座知识，只要让模型学会特定推荐格式、用户建模模板或领域表达，LoRA 已经很够用了。

可能追问：
- 问：Prompt 工程算不算训练？答：通常不算参数训练，它更多是在输入层面引导模型行为。

## 什么是 LLM4Rec？
15秒简答：LLM4Rec 就是把大语言模型用于推荐系统任务，包括用户建模、召回、排序、解释、对话推荐和生成式推荐等方向。

3分钟详答：LLM4Rec 不是单一模型，而是一类研究方向。它的核心出发点是：大模型有很强的语言理解、常识迁移和指令遵循能力，如果把用户行为、物品信息和推荐任务转写成文本，就可能让模型参与推荐。常见用法包括：用 LLM 做用户/物品文本编码；把行为序列文本化后做 next-item 预测；让 LLM 做 zero-shot 或 few-shot ranker；做对话式推荐和理由生成；或者把 LLM 作为推荐系统中的一个增强模块。面试里建议把 LLM4Rec 讲成“不是替代传统推荐，而是扩展推荐表达空间和交互方式”。

可能追问：
- 问：LLM4Rec 一定比传统推荐强吗？答：不一定，在大规模高吞吐召回排序上，传统推荐通常仍然更高效。

## 大模型在推荐里常见的三种用法是什么？
15秒简答：第一做文本理解和特征增强，第二做排序或重排辅助，第三做对话式推荐和推荐解释。

3分钟详答：第一类用法是把大模型当特征工程器，比如总结电影简介、抽取标签、生成 embedding、补充用户画像；第二类用法是做候选排序或重排，让 LLM 结合 richer context 判断哪个 item 更适合当前用户；第三类是交互层，把推荐做成问答或对话，顺带生成推荐理由。面试时这样分类很讨巧，因为既覆盖传统推荐结合方式，也覆盖产品形态创新。对你的项目来说，最现实的是第一类和第二类，而不是直接把整条推荐链路都改成生成式。

可能追问：
- 问：为什么解释生成属于大模型的优势场景？答：因为这类任务依赖语言组织和知识归纳，正是 LLM 的强项。

## LLM 作为 zero-shot ranker 可行吗？
15秒简答：可行，但更适合小候选集、高价值场景；它的优势是泛化和语言理解，短板是成本、时延和排序稳定性。

3分钟详答：一些研究表明，LLM 在把用户偏好和候选 item 都文本化后，可以直接做 zero-shot 或 few-shot ranking。这在冷启动、长尾语义匹配、跨领域迁移上很有潜力，因为大模型自带大量世界知识和语言理解能力。但它的代价也很明显：吞吐低、成本高、结果受 prompt 影响大，而且对长候选列表排序效率很差。所以工业里如果用它，通常是放在小候选集重排、高客单价场景或解释生成场景，而不是直接对全库 item 排序。这个回答和你的项目也很契合，因为你已经有候选集，未来完全可以尝试让 LLM 对小候选做辅助重排。

可能追问：
- 问：为什么说 LLM 更适合小候选重排？答：因为需要把用户和候选文本都送进上下文，候选一多成本就迅速失控。

## 用户和物品序列为什么要“文本化”？
15秒简答：因为 LLM 原生处理文本，文本化能把用户行为、电影属性和上下文组织成模型擅长理解的输入格式。

3分钟详答：传统推荐模型直接处理 id 和数值特征，大模型则更适合读自然语言或结构化文本。所以 LLM4Rec 常把用户历史改写成类似“用户最近喜欢了哪些电影、偏好哪些导演和题材”的文本，把 item 写成标题、类型、导演、简介、标签的描述，再拼成 prompt。这样做的好处是能借用大模型已有的语言知识和推理能力，比如理解“悬疑”和“惊悚”接近、“诺兰”和“烧脑科幻”常相关。缺点是文本化不一定比结构化特征更高效，也可能损失一些精确 id 统计信号。面试里说清这个 trade-off 很重要。

可能追问：
- 问：文本化会丢失什么？答：可能丢失高精度 id 共现统计和一些结构化数值细节。

## LLM4Rec 和双塔 / DIN 这类传统推荐模型怎么比较？
15秒简答：传统模型在效率、吞吐和 id 行为建模上更强，LLM4Rec 在语言理解、冷启动、解释和跨域迁移上更有优势。

3分钟详答：传统推荐模型是为大规模工业检索和排序设计的，擅长处理海量 id、行为日志和高吞吐推断，因此在主链路上依然非常强。LLM4Rec 的优势更多在于它能理解文本、常识和复杂指令，对冷启动 item、对话场景和理由生成特别有帮助。你可以把它们理解为两个不同偏向的工具：传统模型偏“效率与行为统计”，大模型偏“语言与泛化”。真实工业方案更可能是融合，而不是替代。比如用双塔做召回，用 DIN 或 MMoE 做主排序，再让 LLM 处理高价值重排、解释或冷启动增强。

可能追问：
- 问：为什么 LLM4Rec 很难直接替代双塔主召回？答：因为主召回要求极高吞吐和极低延迟，而 LLM 推理成本太高。

## 大模型做推荐时为什么会有顺序感知和位置偏差问题？
15秒简答：因为 LLM 对输入 prompt 的顺序很敏感，候选 item 排列方式本身就可能影响判断结果。

3分钟详答：传统排序模型通常对候选的输入顺序影响较小，但 LLM 在处理长 prompt 时，前后位置、模板形式、提示词措辞都会影响输出。这就带来一个问题：如果你把同样的候选顺序打乱，LLM 给出的排序结果可能变化明显。学术上这常被归类为 order sensitivity 或 position bias。缓解方法包括做候选随机打乱再集成、多轮 pairwise 比较、结构化输出约束或让模型先抽取偏好再排序。面试里如果你能提到“LLM 排序不仅看内容，也受 prompt 编排影响”，会显得你不是只会吹大模型。

可能追问：
- 问：为什么这在推荐里是问题？答：因为排序结果应该尽量由用户-物品匹配决定，而不是由 prompt 排列偶然决定。

## 大模型更适合做召回、排序还是重排？
15秒简答：现阶段更适合小候选集的重排、解释和冷启动增强，不太适合大规模主召回。

3分钟详答：这是非常实用的一题。主召回最强调吞吐和延迟，因此目前更适合双塔、向量检索和轻量召回模型；主排序也要扛较大流量，所以通常还是深度 CTR/排序模型更划算。大模型最有现实价值的地方通常是小候选集重排、高价值 query、复杂语义理解、理由生成和对话交互。你完全可以把这个答案和自己的项目结合起来说：未来如果给当前系统接一个 LLM，我会先把它放在 `semantic_hybrid` 候选之后做小候选辅助重排，而不是替代全部召回和排序。

可能追问：
- 问：为什么冷启动增强也适合大模型？答：因为大模型对文本和常识理解强，能在缺少交互日志时更好利用内容信息。

## 如果把大模型接入你的项目，你会怎么落地？
15秒简答：我会优先做三件事：离线生成更细粒度电影标签、对小候选集做语义重排、给结果生成可解释推荐理由。

3分钟详答：从工程和性价比看，我不会让大模型直接替代现有推荐主链路，而是作为增强模块接入。第一步，用 LLM 离线解析电影简介、评论和标签，生成更细粒度语义标签，提升内容理解。第二步，在 `semantic_hybrid` 召回出的少量候选上，用 LLM 做辅助重排，特别针对冷启动和长尾语义匹配。第三步，在最终结果展示层生成推荐理由，比如“因为你喜欢诺兰和悬疑科幻，所以推荐这部电影”。这样既能发挥 LLM 的语言和泛化优势，又不会让系统成本和延迟爆炸。这个答案非常适合你的项目。

可能追问：
- 问：为什么不优先让 LLM 做主召回？答：因为主召回量级最大，现阶段性价比和时延都不合适。

## 为什么不直接用大模型替代整个推荐系统？
15秒简答：因为推荐系统不只是语言理解，还要高吞吐、低时延、强在线学习和海量 id 匹配，大模型目前很难在这些维度全面取代传统系统。

3分钟详答：这题本质是考你有没有工程常识。大模型当然强，但推荐系统主链路强调的是规模、成本、稳定性和持续反馈学习。传统推荐模型在这些方面已经非常成熟，尤其是海量行为日志、ANN 检索、CTR 预估、多目标优化这些能力。大模型的优势在于语义理解、跨域泛化、解释生成、对话交互和冷启动增强，所以更合理的路线是融合，而不是完全替代。面试里如果你回答得太激进，比如“以后都会被 LLM 替代”，通常反而减分。

可能追问：
- 问：那为什么大家还在研究 LLM4Rec？答：因为它在冷启动、对话推荐、解释生成和高层语义理解上确实打开了新空间。

## 从你这个项目继续往 LLM4Rec 演进，最合理的路线是什么？
15秒简答：先做文本增强和解释生成，再做小候选集 LLM 重排，最后再考虑生成式推荐或对话推荐。

3分钟详答：演进路线最好遵循“低风险、可验证、逐步增益”。第一阶段，把大模型当作离线特征工程器，生成电影标签、风格总结、用户偏好摘要；第二阶段，把大模型接到小候选集重排上，验证它能否提升冷启动和语义匹配；第三阶段，再做推荐理由生成和对话式推荐，把产品层体验提升起来；第四阶段，如果数据和资源足够，再尝试更完整的生成式推荐或 LLM agent 式推荐。这样演进的好处是每一步都有清晰收益点，不会一下子把现有系统推翻重来。面试里这样回答，会显得你很有工程落地意识。

可能追问：
- 问：哪一步最值得优先做实验？答：我会优先做离线语义增强和小候选重排，因为最容易验证、风险也最低。
